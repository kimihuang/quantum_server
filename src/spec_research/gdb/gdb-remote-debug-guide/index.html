    QEMU + GDB 远程调试实战指南¶

    基于 cv_test 嵌入式测试框架，在 QEMU (aarch64) + gdbserver 环境下的远程调试全流程演示。 环境：Host x86_64 ↔ QEMU aarch64 (gdbserver :1234)

    一、调试环境架构¶

    ┌─────────────────────────────────────────────────────────┐
│  Host (x86_64)                                          │
│                                                         │
│  aarch64-none-linux-gnu-gdb ──TCP :1234──┐             │
│       + cv_test (带符号 aarch64)          │             │
│       + sysroot (目标根文件系统)            │             │
│                                          │             │
└──────────────────────────────────────────┼─────────────┘
                                           │
                                    QEMU user-mode NAT
                                    hostfwd=tcp::1234-:1234
                                           │
┌──────────────────────────────────────────┼─────────────┐
│  QEMU (aarch64)                          │             │
│                                          ▼             │
│  gdbserver :1234 /bin/cv_test                          │
│       (监听 1234 端口，等待 GDB 连接)                    │
│                                                         │
│  /mnt/host_shared  ←→  9p virtio 共享目录               │
└─────────────────────────────────────────────────────────┘

    1.1 关键文件路径¶

    用途 路径
    QEMU 启动脚本 src/packages/qemu_sample_env/qemu_gdb_debug.sh
    gdbserver 启动脚本 out/quantum_qemu_debug/qemu_host_shared/start_gdbserver.sh
    GDB 连接脚本 src/packages/qemu_sample_env/gdb.sh
    aarch64 调试符号文件 out/quantum_qemu_debug/build/cv-framework-1.0/cv_test
    目标 sysroot out/quantum_qemu_debug/target/

    1.2 调试通路启动步骤¶

    # Step 1: 启动 QEMU（screen 会话，端口转发 1234）
bash qemu_gdb_debug.sh

# Step 2: 在 QEMU 内启动 gdbserver（通过 screen 发送命令）
screen -S qemu_debug -X stuff '/mnt/host_shared/start_gdbserver.sh\n'

# Step 3: Host 端连接 GDB
aarch64-none-linux-gnu-gdb cv_test   # 使用 aarch64 版本的 cv_test
(gdb) set sysroot /path/to/target
(gdb) target remote localhost:1234

    注意：gdb.sh 默认指向 x86_64 的 cv_test，远程调试时应使用 aarch64 版本： out/quantum_qemu_debug/build/cv-framework-1.0/cv_test（带 debug_info，not stripped）

    4.3 finish — 执行到函数返回¶

    (gdb) break cv_option_parse
(gdb) continue
(gdb) finish     # 执行到 cv_option_parse 返回

    4.4 执行控制命令汇总¶

    命令 缩写 用途
    continue c 继续执行到下一个断点
    stepi [N] si 汇编级单步进入（N 条指令）
    nexti [N] ni 汇编级单步跳过（N 条指令）
    step [N] s 源码级单步进入（已支持）
    next [N] n 源码级单步跳过（已支持）
    finish fin 执行到当前函数返回
    until [line] u 执行到指定行号（跳出循环）

    说明：CMakeLists.txt 已添加 -g -fno-omit-frame-pointer，源码级 step/next/list 均可正常使用。 注意 -O2 优化可能导致部分局部变量被优化掉（显示 <optimized out>），此时可改用 stepi 或在 CMakeLists.txt 中临时将优化级别改为 -Og 以获得最佳调试体验。

    六、寄存器检查¶

    6.1 查看关键寄存器¶

    (gdb) info registers x0 x1 x29 x30 sp pc cpsr

    在 main 入口处的输出： 

    x0             0x1                 1          # argc = 1
x1             0x7ff35eb578        549543916920  # argv 指针
x29            0x7ff35eb400        549543916544  # 帧指针 (FP)
x30            0x7f870db26c        547726668396  # 链接寄存器 (LR)
sp             0x7ff35eb400        0x7ff35eb400  # 栈指针
pc             0x55947c7600        0x55947c7600 <main>  # 程序计数器
cpsr           0x80000000          [ EL=0 BTYPE=0 N ]  # 状态寄存器

    6.2 ARM64 寄存器对照¶

    寄存器 用途 说明
    x0-x7 参数/返回值 函数调用参数和返回值
    x8 间接结果 返回大结构体时指向内存
    x9-x15 临时寄存器 调用者保存
    x16-x17 IP0/IP1 链接器/PLT 临时使用
    x18 平台寄存器 平台特定用途
    x19-x28 被调用者保存 函数需保存恢复
    x29 (FP) 帧指针 指向当前栈帧
    x30 (LR) 链接寄存器 函数返回地址
    sp 栈指针 当前栈顶
    pc 程序计数器 当前执行地址
    cpsr 状态寄存器 条件标志 + EL 等

    8.3 x 命令格式速查¶

    x/NFU addr
  N - 重复次数
  F - 格式: x(十六进制) d(十进制) t(二进制) o(八进制) u(无符号) f(浮点) i(指令) s(字符串) a(地址)
  U - 单位: b(1字节) h(2字节) w(4字节) g(8字节)

    常用示例：

    x/16xw 0x400800       # 16 个 4 字节十六进制
x/20i $pc              # 20 条指令反汇编
x/s (char*)ptr         # 字符串
x/10hg &float_array    # 10 个 2 字节十六进制

    十三、常见问题与解决¶

    Q1: Cannot insert breakpoint / Cannot access memory¶

    原因：断点地址在程序尚未加载的区域（如 gdbserver 刚启动时程序还在动态链接器中）。

    解决：先 break main，再 continue，让动态链接器完成加载后再设断点。

    Q2: 变量显示 <optimized out>¶

    原因：-O2 优化将局部变量分配到寄存器或直接消除，GDB 无法跟踪。

    解决： - 使用 info registers 查看寄存器中的值 - 使用 disassemble /m 从汇编推断变量位置 - 在 CMakeLists.txt 中将 -O2 改为 -Og（专为调试优化的级别）

    Q3: g_framework has unknown type¶

    已解决：添加 -g 后，DWARF 包含完整类型信息，print g_framework 直接输出结构体内容，无需手动 cast。

    Q4: gdbserver 报 "Child exited with status 1"¶

    原因：目标程序正常退出，返回码非 0（测试用例失败）。

    解决：在程序退出前设断点（如 break cv_run），或在 QEMU 内重新启动 gdbserver。

    Q5: static 函数无法设断点（如 run_single_case）¶

    原因：static 函数可能被编译器内联或符号未导出。

    解决：使用地址断点 break *0xADDRESS，或通过 info files / nm 查找函数地址。
