    QEMU + GDB Core Dump 分析实战指南¶

    基于 crash_test 程序，在 QEMU (aarch64) 嵌入式 Linux 环境下生成并分析 core dump 的全流程演示。 环境：Host x86_64 (交叉 GDB) ↔ QEMU aarch64 (目标程序崩溃生成 core)

    一、Core Dump 基础¶

    1.1 什么是 Core Dump¶

    Core dump（核心转储）是程序崩溃时操作系统保存的进程内存映像，包含：

    • 崩溃瞬间的寄存器状态
    • 栈内存（调用栈、局部变量）
    • 堆内存（动态分配的数据）
    • 共享库映射信息

    1.2 Core Dump 分析的价值¶

    场景 价值
    线上崩溃 事后分析崩溃根因，无需复现
    间歇性 bug 捕获崩溃现场，定位竞态/时序问题
    嵌入式设备 现场抓取 core，离线分析
    SIGSEGV/SIGABRT 精确到源码行的崩溃定位

    二、环境配置¶

    2.1 目标端（QEMU 内）启用 Core Dump¶

    # 允许生成任意大小的 core 文件
ulimit -c unlimited

# 查看 core 文件生成模式
cat /proc/sys/kernel/core_pattern
# 输出: core  (表示在当前目录生成名为 "core" 的文件)

# 创建 core 存放目录
mkdir -p /tmp/cores
cd /tmp/cores

    2.2 core_pattern 说明¶

    # 默认: 当前目录生成 "core"
core

# 带格式（推荐生产环境使用）
echo "/var/core/core.%e.%p.%t" > /proc/sys/kernel/core_pattern
# %e - 程序名
# %p - PID
# %t - Unix 时间戳
# %s - 触发信号

    2.3 Host 端准备¶

    # 交叉 GDB
GDB=/path/to/aarch64-none-linux-gnu-gdb

# 带调试符号的二进制文件（-g -O0 编译）
BINARY=/path/to/build/crash-test-1.0/crash_test

# 目标 sysroot
SYSROOT=/path/to/target

# Core 文件（从 QEMU 拷贝到 host）
CORE=/path/to/core_null_deref

    2.4 crash_test 程序¶

    crash_test 包含 8 种崩溃场景，使用 -g -O0 编译以获得最佳调试体验：

    场景 参数 崩溃类型 信号
    1 crash_test 1 空指针解引用 SIGSEGV
    2 crash_test 2 缓冲区越界写入 SIGSEGV
    3 crash_test 3 非法地址访问 SIGSEGV
    4 crash_test 4 abort() SIGABRT
    5 crash_test 5 Use-After-Free + double-free SIGABRT
    6 crash_test 6 栈溢出 SIGSEGV
    7 crash_test 7 整数除零 SIGFPE
    8 crash_test 8 自定义信号处理 + crash SIGSEGV

    三、Core Dump 生成与拷贝¶

    3.1 运行崩溃程序¶

    # 在 QEMU 内
cd /tmp/cores
rm -f core                  # 清除旧 core 文件
crash_test 1                # 运行场景1: 空指针解引用
# 输出: Segmentation fault (core dumped)
# 退出码: 139 (128 + 11 = SIGSEGV)

    3.2 拷贝 Core 文件到 Host¶

    # 在 QEMU 内: 拷贝到 9p 共享目录
cp /tmp/cores/core /mnt/host_shared/core_null_deref

    3.3 退出码与信号对照¶

    退出码 信号 信号编号
    139 SIGSEGV 11
    134 SIGABRT 6
    136 SIGFPE 8
    135 SIGBUS 7

    四、Core Dump 分析实战¶

    场景 1：空指针解引用 (SIGSEGV)¶

    启动 GDB 加载 core¶

    aarch64-none-linux-gnu-gdb crash_test core_null_deref

    (gdb) set sysroot /path/to/target
(gdb) core-file core_null_deref
[New LWP 127]
Core was generated by `crash_test 1'.
Program terminated with signal SIGSEGV, Segmentation fault.
#0  0x0000005575083a30 in crash_null_deref () at crash_test.c:14
14      int val = *ptr;  /* SIGSEGV */

    GDB 直接定位到崩溃源码行：crash_test.c:14

    调用栈分析¶

    (gdb) bt full

    #0  0x0000005575083a30 in crash_null_deref () at crash_test.c:14
        ptr = 0x0                ← ptr 是 NULL!
        val = 127
#1  0x0000005575083d9c in main (argc=2, argv=0x7fc295f6b8) at crash_test.c:98
        scenario = 1

    结论：ptr = 0x0（空指针），解引用 *ptr 导致 SIGSEGV。

    信号详细信息¶

    (gdb) print $_siginfo

    $1 = {
  si_signo = 11,           ← SIGSEGV
  si_code = 1,             ← SEGV_MAPERR (地址未映射)
  _sifields = {
    _sigfault = {
      si_addr = 0x0        ← 故障地址: NULL (0x0)
    }
  }
}

    解读： - si_signo = 11 → SIGSEGV - si_code = 1 → SEGV_MAPERR，表示访问的地址没有映射到任何内存区域 - si_addr = 0x0 → 故障地址是 NULL，确认是空指针解引用

    寄存器检查¶

    (gdb) info registers x0 x1 x29 x30 sp pc

    x0             0x0                 0         ← x0=0, 正是 NULL ptr 的值
x29            0x7fc295f4f0        ...       ← 帧指针
x30            0x5575083a2c        ...       ← 返回地址
sp             0x7fc295f4f0        ...
pc             0x5575083a30        <crash_null_deref+28>  ← 崩溃指令

    反汇编崩溃函数¶

    (gdb) disassemble /m crash_null_deref

    11  {
   0x5575083a14 <+0>:   stp  x29, x30, [sp, #-32]!
   0x5575083a18 <+4>:   mov  x29, sp

12      int *ptr = NULL;
   0x5575083a1c <+8>:   str  xzr, [sp, #24]         ← ptr = NULL (xzr = zero register)

13      printf("[场景1] 空指针解引用: *NULL ...\n");
   0x5575083a20 <+12>:  adrp x0, 0x5575083000
   0x5575083a24 <+16>:  add  x0, x0, #0xea0
   0x5575083a28 <+20>:  bl   puts@plt

14      int val = *ptr;  /* SIGSEGV */
   0x5575083a2c <+24>:  ldr  x0, [sp, #24]          ← 加载 ptr 值到 x0 (=0x0)
=> 0x5575083a30 <+28>:  ldr  w0, [x0]               ← 从地址 0x0 加载 → SIGSEGV!
   0x5575083a34 <+32>:  str  w0, [sp, #20]

    场景 3：非法地址访问 (SIGSEGV)¶

    (gdb) core-file core_invalid_addr
[New LWP 138]
Core was generated by `crash_test 3'.
Program terminated with signal SIGSEGV, Segmentation fault.
#0  0x000000558b6daad4 in crash_invalid_addr () at crash_test.c:31
31      *ptr = 42;  /* SIGSEGV: 写入未映射地址 */

(gdb) bt full
#0  crash_invalid_addr () at crash_test.c:31
        ptr = 0xdeadbeef         ← 指向非法地址
#1  main (argc=2, argv=0x7fc023aba8) at crash_test.c:100
        scenario = 3

(gdb) print $_siginfo._sifields._sigfault.si_addr
$1 = (void *) 0xdeadbeef         ← 故障地址确认是 0xDEADBEEF

    分析：ptr = 0xdeadbeef，向未映射地址写入 42 导致 SIGSEGV。si_addr 确认故障地址。

    场景 4：abort() (SIGABRT)¶

    (gdb) core-file core_abort
[New LWP 132]
Core was generated by `crash_test 4'.
Program terminated with signal SIGABRT, Aborted.
#0  0x0000007fb63d2800 in ?? () from libc.so.6

(gdb) bt full
#0  0x0000007fb63d2800 in ?? () from libc.so.6       ← 内核信号处理
#1  raise () from libc.so.6
#2  abort () from libc.so.6                          ← abort() 调用 raise(SIGABRT)
#3  crash_abort () at crash_test.c:38                ← 用户代码: abort()
#4  main (argc=2, argv=0x7ff962f838) at crash_test.c:101
        scenario = 4

(gdb) print $_siginfo.si_signo
$1 = 6                ← SIGABRT

(gdb) print $_siginfo.si_code
$2 = -6               ← SI_TKILL (由 tkill/sendtid 发送)

    分析：abort() → raise(SIGABRT) → 内核信号处理。用户代码在 frame 3，切换栈帧查看源码：

    (gdb) frame 3
#3  crash_abort () at crash_test.c:38
38      abort();

(gdb) list
34  /* ====== 场景4: abort (SIGABRT) ====== */
35  static void crash_abort(void)
36  {
37      printf("[场景4] 调用 abort() ...\n");
38      abort();
39  }

    场景 5：Use-After-Free + Double-Free (SIGABRT)¶

    (gdb) core-file core_uaf_doublefree
[New LWP 135]
Core was generated by `crash_test 5'.
Program terminated with signal SIGABRT, Aborted.

(gdb) bt full
#0  ?? () from libc.so.6
#1  raise () from libc.so.6
#2  abort () from libc.so.6
#3  ?? () from libc.so.6              ← malloc 内部检测逻辑
#4  ?? () from libc.so.6
#5  ?? () from libc.so.6
#6  free () from libc.so.6            ← 崩溃在 free() 中
#7  crash_use_after_free () at crash_test.c:50
        ptr = 0x55768162b0            ← 已被释放的指针
#8  main (argc=2, argv=0x7fd75deef8) at crash_test.c:102
        scenario = 5

(gdb) frame 7
#7  crash_use_after_free () at crash_test.c:50
50      free(ptr);                    ← 第二次 free(ptr) → double-free!

    分析：glibc 的 tcache 机制检测到 double-free，调用 abort() 终止程序。调用栈中 free() → glibc 内部检测 → abort() 是典型的 double-free 崩溃模式。

    QEMU 侧运行输出印证： 

    [场景5] Use-After-Free: 访问已释放内存 ...
  已释放 ptr=0x55768162b0, 现在读取 *ptr = 89614358   ← UAF: 读取到脏数据
free(): double free detected in tcache 2               ← glibc 检测到 double-free
Aborted (core dumped)

    场景 6：栈溢出 (SIGSEGV)¶

    (gdb) core-file core_stack_overflow
[New LWP 141]
Core was generated by `crash_test 6'.
Program terminated with signal SIGSEGV, Segmentation fault.
#0  0x0000007fb8c47210 in ?? () from libc.so.6

(gdb) bt 30
#0  ?? () from libc.so.6                            ← 崩溃在 printf 内部
#1  ?? () from libc.so.6
#2  printf () from libc.so.6                        ← printf 需要栈空间，栈已耗尽
#3  crash_stack_overflow (depth=2022) at crash_test.c:59
#4  crash_stack_overflow (depth=2021) at crash_test.c:60
#5  crash_stack_overflow (depth=2020) at crash_test.c:60
...
#29 crash_stack_overflow (depth=1996) at crash_test.c:60

    分析： - 递归调用 crash_stack_overflow 2022 层，每层分配 4096 字节，栈空间耗尽 - 崩溃发生在 printf() 中（printf 需要栈缓冲区，但栈已满） - 识别特征：调用栈中大量重复的同一函数 + depth 参数递增

    五、Core Dump 分析标准流程¶

    5.1 快速三步法¶

    Step 1: 加载 core → 确认信号和崩溃位置
Step 2: bt full → 查看完整调用栈和变量
Step 3: print $_siginfo → 确认故障地址

    5.2 详细分析流程¶

    1. 加载 core dump
   (gdb) set sysroot /path/to/target
   (gdb) core-file core_file

2. 确认崩溃信号
   (gdb) info program          # 崩溃信号
   (gdb) print $_siginfo       # 信号详情

3. 查看调用栈
   (gdb) bt                    # 调用栈
   (gdb) bt full               # 调用栈 + 所有局部变量

4. 分析崩溃帧
   (gdb) frame 0               # 切换到崩溃帧
   (gdb) list                  # 查看源码
   (gdb) info locals           # 局部变量
   (gdb) info args             # 函数参数

5. 检查关键变量
   (gdb) print ptr             # 指针值
   (gdb) print *ptr            # 解引用（如果是有效地址）
   (gdb) print $_siginfo._sifields._sigfault.si_addr  # 故障地址

6. 寄存器检查
   (gdb) info registers        # 所有寄存器
   (gdb) info registers x0 x30 pc sp  # 关键寄存器

7. 反汇编确认
   (gdb) disassemble /m        # 混合源码和汇编

    六、信号与故障地址速查¶

    6.1 SIGSEGV (信号 11) 的 si_code¶

    si_code 宏名 含义
    1 SEGV_MAPERR 地址未映射（空指针、非法地址）
    2 SEGV_ACCERR 权限不足（写只读内存等）

    6.2 si_addr（故障地址）解读¶

    si_addr 值 含义
    0x0 空指针解引用
    0xDEADBEEF 等 野指针/未初始化指针
    栈附近地址 可能栈溢出
    堆附近地址 可能堆越界/UAF

    6.3 常见崩溃模式识别¶

    调用栈特征 崩溃类型
    崩溃在用户代码，ptr=NULL 空指针解引用
    崩溃在 free()，之前有 free 同一指针 double-free
    崩溃在 printf/memcpy，栈上大量递归帧 栈溢出
    崩溃在用户代码，si_addr 为野地址 非法地址访问
    崩溃在 abort()，调用链有 malloc/free 堆损坏/双重释放
    崩溃在 abort()，用户代码直接调用 断言失败/主动 abort

    七、GDB 批处理分析 Core Dump¶

    aarch64-none-linux-gnu-gdb -batch \
  -ex "set pagination off" \
  -ex "set print pretty on" \
  -ex "set sysroot /path/to/target" \
  -ex "core-file /path/to/core_file" \
  -ex "echo === Signal ===\n" \
  -ex "print \$_siginfo.si_signo" \
  -ex "print \$_siginfo._sifields._sigfault.si_addr" \
  -ex "echo === Backtrace ===\n" \
  -ex "bt full" \
  -ex "echo === Registers ===\n" \
  -ex "info registers x0 x1 x29 x30 sp pc" \
  -ex "echo === Disassembly ===\n" \
  -ex "frame 0" \
  -ex "disassemble /m" \
  -ex "quit" \
  /path/to/crash_test

    八、嵌入式 Core Dump 最佳实践¶

    8.1 编译选项¶

    # CMakeLists.txt 推荐配置
add_compile_options(-g -O0 -fno-omit-frame-pointer)
    • -g：生成 DWARF 调试信息
    • -O0：不优化，变量不被优化掉
    • -fno-omit-frame-pointer：保留帧指针，确保调用栈可回溯

    8.2 目标端配置¶

    # /etc/profile 或 init 脚本
ulimit -c unlimited

# 自定义 core 文件名（包含程序名和PID）
echo "/var/core/core.%e.%p.%t" > /proc/sys/kernel/core_pattern

# 确保 core 目录存在且有写权限
mkdir -p /var/core && chmod 1777 /var/core

    8.3 Core 文件收集流程¶

    1. 嵌入式设备崩溃 → 生成 core 文件
2. 通过 9p/SCP/TFTP 拷贝到开发主机
3. 使用交叉 GDB 加载分析

    8.4 Core 文件精简（存储受限设备）¶

    # 仅保存必要的内存区域（减小 core 文件大小）
echo 0x3f > /proc/self/coredump_filter
# bit 0: 匿名私有内存 (必须)
# bit 1: 匿名共享内存
# bit 2: 文件映射私有内存
# bit 3: 文件映射共享内存
# bit 4: ELF 头
# bit 5: 大页私有内存

    九、常见问题¶

    Q1: "warning: Can't open file /path/lib.so" 加载 core 时¶

    解决：设置 sysroot 

    (gdb) set sysroot /path/to/target/rootfs

    Q2: 调用栈显示 ?? () 无符号信息¶

    原因： - 共享库缺少调试符号（stripped） - 使用了错误的 binary（如 x86_64 而非 aarch64）

    解决：确保使用正确的带符号 aarch64 binary

    Q3: Core 文件为空或未生成¶

    原因： - ulimit -c 限制为 0 - core_pattern 管道到其他程序（如 apport） - 磁盘空间不足

    解决： 

    ulimit -c unlimited
cat /proc/sys/kernel/core_pattern   # 检查设置
df -h                               # 检查磁盘空间

    Q4: bt full 中变量显示 <optimized out>¶

    原因：编译时开启了优化（-O1/-O2）

    解决：使用 -O0 -g -fno-omit-frame-pointer 重新编译

    Q5: 如何区分空指针和非法地址？¶

    查看 $_siginfo._sifields._sigfault.si_addr： - si_addr = 0x0 → 空指针解引用 - si_addr = 0xdeadbeef 等非零非法值 → 野指针/未初始化指针

    附录：Core Dump 分析命令速查¶

    步骤 命令 用途
    加载 core-file /path/to/core 加载 core dump
    信号 print $_siginfo 崩溃信号和故障地址
    调用栈 bt full 完整调用栈 + 变量
    栈帧 frame N / up / down 切换栈帧
    变量 info locals / print var 查看变量
    源码 list 查看崩溃源码
    反汇编 disassemble /m func 混合源码和汇编
    寄存器 info registers 崩溃时寄存器
    故障地址 print $_siginfo._sifields._sigfault.si_addr SIGSEGV 故障地址
    内存 x/16xw addr 查看内存内容
    批处理 gdb -batch -ex "bt full" ... 自动化分析
