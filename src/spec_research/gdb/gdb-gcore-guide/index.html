    GDB gcore 运行时 Core Dump 实战指南¶

    基于 loop_test 多线程程序，在 QEMU (aarch64) 环境下演示 gcore 生成运行时 core dump 并离线分析。 与崩溃 core dump 不同，gcore 可以在不终止进程的情况下抓取运行时快照。

    一、gcore 与崩溃 Core Dump 的区别¶

    特性 崩溃 Core Dump gcore 运行时 Dump
    触发方式 程序崩溃（SIGSEGV/SIGABRT） 手动执行 gcore 命令
    进程状态 已终止 仍在运行
    适用场景 事后分析崩溃原因 分析运行时状态、卡死、性能问题
    数据内容 崩溃瞬间的内存映像 执行 gcore 时刻的内存映像
    对业务影响 进程已死，需重启 进程暂停后恢复，短暂影响

    gcore 典型使用场景¶

    • 生产环境卡死：进程 hang 住但不崩溃，gcore 抓取现场分析
    • 性能问题：抓取运行时数据，分析哪个函数占 CPU
    • 间歇性 bug：在问题出现时手动抓取，无需复现
    • 多线程死锁：抓取所有线程栈，分析锁等待关系
    • 内存泄漏：多次 gcore 对比，查看内存增长

    二、环境准备¶

    2.1 目标端启动长期运行进程¶

    # 在 QEMU 内后台运行 loop_test
loop_test &

# 确认进程 PID
ps | grep loop_test
#   129 root     loop_test

    2.2 loop_test 程序结构¶

    // 全局状态
static int g_counter = 0;              // 计数器
static char g_message[256];            // 状态消息
static work_item_t g_work_queue[8];    // 工作队列
static int g_queue_len = 0;            // 队列长度
static int g_running = 1;              // 运行标志

// 3 个线程:
// Thread 1 (main):    主循环 sleep(5) + 打印状态
// Thread 2 (worker1): 每 2s 递增 counter + 更新 message
// Thread 3 (worker2): 每 2s 递增 counter + 更新 message

    2.3 关键文件路径¶

    项目 路径
    loop_test 源码 src/packages/loop_test/loop_test.c
    带符号 binary out/quantum_qemu_debug/build/loop-test-1.0/loop_test
    目标 sysroot out/quantum_qemu_debug/target/
    交叉 GDB out/quantum_qemu_debug/host/bin/aarch64-none-linux-gnu-gdb

    三、gcore 生成 Core Dump¶

    3.1 方法一：Host GDB 远程 gcore（推荐嵌入式方案）¶

    步骤 1：目标端 gdbserver attach 到进程

    # 在 QEMU 内，attach 到 PID 129
gdbserver --attach :1234 129 &
# Attached; pid = 129
# Listening on port 1234

    步骤 2：Host GDB 连接 + gcore

    (gdb) set sysroot /path/to/target
(gdb) target remote localhost:1234

# 查看线程信息
(gdb) info threads
  Id   Target Id                  Frame
* 1    Thread 129.129 "loop_test" 0x7f808242f0 in clock_nanosleep ()
  2    Thread 129.130 "loop_test" 0x7f808242f0 in clock_nanosleep ()
  3    Thread 129.131 "loop_test" 0x7f808242f0 in clock_nanosleep ()

# 生成 core dump 到 host
(gdb) gcore /path/to/core_loop_test_gcore
Saved corefile /path/to/core_loop_test_gcore

    步骤 3：detach（进程继续运行）

    (gdb) detach
[Inferior 1 (process 129) detached]

(gdb) quit

    步骤 4：验证进程仍在运行

    # 在 QEMU 内
ps | grep loop_test
#   129 root     loop_test    ← 进程仍在运行！

    关键：detach 让进程恢复执行，gcore 只短暂暂停进程，不影响后续运行。

    3.2 方法二：目标端 gcore（需要 bash）¶

    # 在 QEMU 内（需要 bash 环境）
gcore <PID>
# 生成 core.<PID> 到当前目录

    注意：Buildroot 默认只有 ash/sh，gcore 脚本依赖 bash。如需此方式， 需在 Buildroot 中启用 bash，或使用方法一（Host GDB 远程 gcore）。

    3.3 gcore 命令详解¶

    # 基本用法
(gdb) gcore <filename>              # 生成 core dump 到指定文件

# 指定格式
(gdb) gcore /tmp/core_%p            # %p = PID

# 只保存特定线程的内存（减少文件大小）
(gdb) set coredump-filter 0x3f      # 所有内存区域
(gdb) gcore /tmp/core_dump

# gcore 后进程自动恢复执行
# 无需额外操作

    四、离线分析 gcore Core Dump¶

    4.1 加载 core 文件¶

    aarch64-none-linux-gnu-gdb loop_test core_loop_test_gcore

    (gdb) set sysroot /path/to/target
(gdb) core-file core_loop_test_gcore
[New LWP 129]
[New LWP 130]
[New LWP 131]
Core was generated by `loop_test'.
#0  0x0000007f808242f0 in clock_nanosleep () from libc.so.6

    注意：gcore 生成的 core 文件没有信号信息（进程未崩溃），但包含完整的线程和内存数据。

    4.2 线程分析¶

    (gdb) info threads
  Id   Target Id         Frame
* 1    LWP 129           0x7f808242f0 in clock_nanosleep () from libc.so.6
  2    LWP 130           0x7f808242f0 in clock_nanosleep () from libc.so.6
  3    LWP 131           0x7f808242f0 in clock_nanosleep () from libc.so.6

    3 个线程（1 主线程 + 2 worker），全部在 clock_nanosleep（sleep 的底层实现）。

    4.3 所有线程调用栈¶

    (gdb) thread apply all bt

    Thread 3 (LWP 131):
#0  clock_nanosleep () from libc.so.6
#1  nanosleep () from libc.so.6
#2  sleep () from libc.so.6
#3  worker_thread (arg=0x7fd0d397a0) at loop_test.c:33    ← worker 线程
#4  ?? () from libc.so.6                                    ← pthread_create

Thread 2 (LWP 130):
#0  clock_nanosleep () from libc.so.6
#1  nanosleep () from libc.so.6
#2  sleep () from libc.so.6
#3  worker_thread (arg=0x7fd0d3979c) at loop_test.c:33    ← worker 线程
#4  ?? () from libc.so.6

Thread 1 (LWP 129):
#0  clock_nanosleep () from libc.so.6
#1  nanosleep () from libc.so.6
#2  sleep () from libc.so.6
#3  main () at loop_test.c:66                               ← 主线程

    解读： - Thread 1（主线程）：main → sleep，在主循环的 sleep(5) 中 - Thread 2（worker1）：worker_thread → sleep，在 sleep(2) 中 - Thread 3（worker2）：worker_thread → sleep，在 sleep(2) 中

    4.4 全局变量分析¶

    (gdb) print g_counter
$1 = 36

(gdb) print g_message
$2 = "thread_1 working, counter=36", '\000' <repeats 227 times>

(gdb) print g_queue_len
$3 = 5

(gdb) print g_running
$4 = 1               ← 进程仍在正常运行

    4.6 切换线程查看局部变量¶

    # 切换到 Thread 2 (worker)
(gdb) thread 2

# 查看完整调用栈
(gdb) bt full
#3  worker_thread (arg=0x7fd0d3979c) at loop_test.c:33
        thread_id = 1           ← worker 线程 ID

# 切换到 worker_thread 帧
(gdb) frame 3
#3  worker_thread (arg=0x7fd0d3979c) at loop_test.c:33
33          sleep(2);

# 查看源码
(gdb) list
28  static void *worker_thread(void *arg)
29  {
30      int thread_id = *(int *)arg;
31      while (g_running) {
32          sleep(2);
33          g_counter++;
34          snprintf(g_message, BUF_SIZE,
35                   "thread_%d working, counter=%d", thread_id, g_counter);
36      }
37      return NULL;
38  }

# 查看线程参数
(gdb) print *(int *)arg
$6 = 1               ← arg 指向的值为 1 (id1=1)

    五、gcore 实战场景¶

    场景 1：分析多线程死锁¶

    # 1. 发现进程卡死，gcore 抓取现场
gdbserver --attach :1234 <PID> &
# Host:
(gdb) target remote localhost:1234
(gdb) gcore /tmp/core_deadlock
(gdb) detach

# 2. 离线分析
(gdb) core-file /tmp/core_deadlock
(gdb) thread apply all bt
# 找到所有线程停在 __lll_lock_wait / pthread_mutex_lock
# 分析锁链: Thread A 持有 mutex1 等待 mutex2, Thread B 持有 mutex2 等待 mutex1

    场景 2：多次 gcore 对比内存增长¶

    # 第一次 gcore
(gdb) gcore /tmp/core_snapshot_1
# 等待 10 分钟
# 第二次 gcore
(gdb) gcore /tmp/core_snapshot_2

# 对比
(gdb) core-file /tmp/core_snapshot_1
(gdb) call malloc_stats()
(gdb) core-file /tmp/core_snapshot_2
(gdb) call malloc_stats()
# 比较两次的内存分配统计

    场景 3：生产环境远程抓取¶

    # 目标设备（串口/SSH）
gdbserver --attach :1234 <hung_pid> &

# 开发主机
aarch64-none-linux-gnu-gdb ./app
(gdb) set sysroot /path/to/target
(gdb) target remote <device_ip>:1234
(gdb) gcore ./core_hung_process
(gdb) detach
(gdb) quit
# 进程恢复正常，core 文件带回分析

    六、gcore 与崩溃 Core Dump 对比分析¶

    分析项 崩溃 Core Dump gcore Core Dump
    加载方式 core-file <path> core-file <path>
    信号信息 $_siginfo 有效（SIGSEGV/SIGABRT） 无信号（进程未崩溃）
    线程信息 info threads 有效 info threads 有效
    调用栈 bt full 有效 bt full 有效
    全局变量 print var 有效 print var 有效
    局部变量 info locals 有效 info locals 有效
    进程状态 已终止 快照时刻的运行状态

    八、常见问题¶

    Q1: 目标端 gcore 报 "env: can't execute 'bash'"¶

    原因：gcore 脚本依赖 bash，Buildroot 默认只有 ash/sh。

    解决：使用 Host GDB 远程 gcore（方法一），或在 Buildroot 中启用 bash。

    Q2: gcore 后进程是否受影响？¶

    影响：gcore 执行期间进程短暂暂停（通常几秒），core 写完后自动恢复。对时间敏感的实时系统需注意。

    Q3: gcore 文件很大，如何精简？¶

    # 设置 core dump 过滤器（减少内存区域）
(gdb) set coredump-filter 0x3f    # 所有区域
(gdb) set coredump-filter 0x01    # 仅匿名私有内存（最小）
(gdb) gcore /tmp/core_small

    Q4: 如何在生产环境自动化 gcore？¶

    #!/bin/sh
# 定时 gcore 脚本
PID=$(pidof my_app)
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
gdbserver --attach :1234 $PID &
sleep 1
aarch64-none-linux-gnu-gdb -batch \
  -ex "set sysroot /path/to/target" \
  -ex "target remote localhost:1234" \
  -ex "gcore /tmp/core_${TIMESTAMP}" \
  -ex "detach" \
  -ex "quit" \
  /path/to/my_app
kill %1 2>/dev/null

    Q5: gcore 和崩溃 core dump 可以用同样的方式分析吗？¶

    可以。两者都是 ELF 格式的 core 文件，加载和分析命令完全一致。唯一的区别是崩溃 core dump 有 $_siginfo 信号信息，gcore 生成的没有。
