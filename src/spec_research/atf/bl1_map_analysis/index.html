    TF-A v2.12 BL1 Linker Map 文件分析报告¶

    1. 概述¶

    • 固件版本: Arm Trusted Firmware-A (TF-A) v2.12
    • 目标平台: QEMU (ARM AArch64)
    • 构建配置: release
    • 目标文件: bl1.elf (elf64-littleaarch64)

    BL1 是 TF-A 启动链的第一个阶段，运行在 EL3 异常级别，负责平台早期初始化并加载 BL2 镜像。

    2. 内存配置 (Memory Configuration)¶

    BL1 的链接脚本定义了两个内存区域：ROM（只读）和 RAM（可读写），运行时 .data 段从 ROM 搬运到 RAM 中执行。

    名称 起始地址 长度 属性 说明
    ROM 0x0000000000000000 0x20000 (128 KB) xr 存放 .text、.rodata 和 .data 的 ROM 副本
    RAM 0x000000000e0ee000 0x12000 (72 KB) xrw 存放运行时 .data、.stacks、.bss 和页表

    2.1 链接器关键符号¶

    符号 值 说明
    ROM_REGION_START 0x0000000000000000 ROM 区域起始地址
    ROM_REGION_END 0x00000000000069f0 ROM 区域结束地址
    RAM_REGION_START 0x000000000e0ee000 RAM 区域起始地址
    RAM_REGION_END 0x000000000e0f6000 RAM 区域结束地址
    __BL1_ROM_END__ 0x0000000000006af5 ROM 有效数据结束（含 .data 的 ROM 拷贝）
    __BL1_RAM_START__ 0x000000000e0ee000 BL1 RAM 起始
    __BL1_RAM_END__ 0x000000000e0f6000 BL1 RAM 结束
    __BSS_SIZE__ 0x8a0 (2208 B) BSS 段大小
    __DATA_SIZE__ 0x105 (261 B) .data 段大小
    __TEXT_START__ 0x0000000000000000 代码段起始
    __TEXT_END__ 0x0000000000005000 代码段结束（页对齐后）
    __RODATA_START__ 0x0000000000005000 只读数据段起始
    __RODATA_END__ 0x00000000000069e8 只读数据段结束
    __RW_START__ 0x000000000e0ee000 读写区域起始
    __RW_END__ 0x000000000e0f6000 读写区域结束
    __DATA_ROM_START__ 0x00000000000069f0 .data 在 ROM 中的 load address
    __CPU_OPS_START__ 0x00000000000066e8 CPU 操作表起始
    __CPU_OPS_END__ 0x00000000000069e8 CPU 操作表结束
    __XLAT_TABLE_START__ 0x000000000e0f0000 MMU 页表段起始
    __XLAT_TABLE_END__ 0x000000000e0f6000 MMU 页表段结束
    __STACKS_START__ 0x000000000e0ee140 栈段起始
    __STACKS_END__ 0x000000000e0ef140 栈段结束

    2.2 空描述符区域（均为空）¶

    以下链接器描述符区域在 BL1 中大小为 0：

    描述符区域 地址 说明
    __RT_SVC_DESCS_START__ / __END__ 0x66E8 运行时服务描述符（BL1 无 RT svc）
    __FCONF_POPULATOR_START__ / __END__ 0x66E8 FCONF 动态配置
    __PMF_SVC_DESCS_START__ / __END__ 0x66E8 PMF 服务描述符
    __PARSER_LIB_DESCS_START__ / __END__ 0x66E8 镜像解析器库描述符
    __BAKERY_LOCK_START__ / __END__ 0xE0EF9C0 Bakery Lock（单核无需锁）
    __PMF_TIMESTAMP_START__ / __END__ 0xE0EF9C0 PMF 时间戳（未启用）
    .got (__GOT_START__ / __END__) 0x69E8 全局偏移表（静态链接无需 GOT）
    .iplt 0x5000 间接 PLT（未使用）
    .rela.dyn 0x69F0 运行时重定位（未使用）

    3. Section 布局详解¶

    3.1 总体内存布局图¶

    ROM 区域 (0x0 - 0x20000, 128KB)
┌─────────────────────────────────────────────┐
│ 0x0000  .text 代码段                        │ 0x5000 (20 KB)
│  ┌─ 0x0000  bl1_entrypoint (入口点)         │
│  │  0x0100  bl1_arch_setup                  │
│  │  0x01D4  bl1_setup                       │
│  │  0x01E8  bl1_main (主函数)               │
│  │  ...                                     │
│  └─ 0x4FFF  (代码段结束)                    │
├─────────────────────────────────────────────┤
│ 0x5000  .rodata 只读数据段                   │ 0x19F0 (6640 B)
│  ├─ IO 设备操作函数表 (fip/memmap/sh)        │
│  ├─ plat_qemu_mmap (QEMU 内存映射表)         │
│  ├─ errata_entries (CPU勘误表, 7种核心)      │
│  ├─ build_version_string / build_message    │
│  ├─ UUID specs (BL2~BL33 镜像)              │
│  └─ .cpu_ops (CPU 操作表, 11种核心)          │
├─────────────────────────────────────────────┤
│ 0x69F0  .data 的 ROM 拷贝 (load address)    │ 0x105 (261 B)
│         运行时搬运到 RAM: 0x0E0EE000         │
├─────────────────────────────────────────────┤
│ 0x6AF5  ─── BL1 ROM 有效结束 ───             │
├─────────────────────────────────────────────┤
│ 0x6AF5 ~ 0x20000  未使用 ROM 空间             │
│         (剩余 ~110 KB)                       │
└─────────────────────────────────────────────┘

RAM 区域 (0xE0EE000 - 0xE0F6000, 72KB)
┌─────────────────────────────────────────────┐
│ 0xE0EE000  .data 初始化数据段                 │ 0x105 (261 B)
│  ├─ memmap_dev_info                         │
│  ├─ sh_dev_info                             │
│  ├─ tf_xlat_ctx (XLAT 上下文)               │
│  ├─ bl2_img_desc.0 (BL2 镜像描述)           │
│  ├─ max_log_level                           │
│  └─ console_state                           │
├─────────────────────────────────────────────┤
│ 0xE0EE140  .stacks 栈段                     │ 0x1000 (4 KB)
│  └─ tzfw_normal_stacks                      │
├─────────────────────────────────────────────┤
│ 0xE0EF140  .bss 未初始化数据段               │ 0x8A0 (2208 B)
│  ├─ bl1_cpu_context.0 (CPU 上下文, 1152 B)  │
│  ├─ bl1_cpu_context_ptr                     │
│  ├─ console_list                            │
│  ├─ IO 子系统数据结构                        │
│  ├─ tf_mmap (内存映射描述, 560 B)           │
│  ├─ per_world_context                       │
│  └─ base_xlat_table (基础页表, 32 B)        │
├─────────────────────────────────────────────┤
│ 0xE0F0000  .xlat_table 页表段                │ 0x6000 (24 KB)
│  └─ 运行时 MMU 页表 (translation tables)     │
├─────────────────────────────────────────────┤
│ 0xE0F6000  ─── BL1 RAM 结束 ───              │
└─────────────────────────────────────────────┘

    注意: .text 段中 0x0000~0x37F4 为 C/汇编代码，0x3168~0x3814 为 libc 函数，0x4000~0x47FF 为 EL3 异常向量表（.vectors, 2KB），中间有填充对齐。

    3.2 .text 代码段 (0x0000 - 0x4FFF, 20 KB)¶

    代码段是 BL1 最大的 section，包含所有可执行代码。bl1_entrypoint 位于最开头，确保复位后首先执行。

    3.2.1 BL1 核心启动函数¶

    符号 地址 大小 来源文件 说明
    bl1_entrypoint 0x0000 0x100 (256 B) bl1_entrypoint.o BL1 入口点（汇编），CPU 复位后第一条指令
    bl1_arch_setup 0x0100 0x10 (16 B) bl1_arch_setup.o 架构相关初始化
    cm_get_context 0x0110 0x10 (16 B) bl1_context_mgmt.o 获取 CPU 上下文
    bl1_prepare_next_image 0x0120 0xB4 (180 B) bl1_context_mgmt.o 准备下一个镜像的入口点信息
    bl1_setup 0x01D4 0x14 (20 B) bl1_main.o BL1 设置（公共层）
    bl1_main 0x01E8 0x104 (260 B) bl1_main.o BL1 主函数（C 入口）
    bl1_print_next_bl_ep_info 0x02EC 0x2C (44 B) bl1_main.o 打印下一 BL 阶段入口信息
    bl1_smc_handler 0x0318 0x64 (100 B) bl1_main.o SMC 中断处理函数

    3.2.8 CPU Errata 处理（代码段占比大）¶

    代码段后半部分（0x2084 ~ 0x4FFF）主要由 CPU errata 处理函数和 CPU reset 函数占据，支持以下核心：

    核心类型 代码范围 说明
    Cortex-A53 0x2420 - 0x2517 8 个 erratum 检查 + reset
    Cortex-A55 0x2518 - 0x25B7 6 个 erratum 检查 + reset
    Cortex-A57 0x25B8 - 0x272F 14 个 erratum 检查 + reset
    Cortex-A710 0x2738 - 0x2837 16 个 erratum 检查 + reset
    Cortex-A72 0x2838 - 0x295F 7 个 erratum 检查 + reset
    Cortex-A76 0x2960 - 0x2A2F 12 个 erratum 检查 + reset
    Neoverse-N1 0x2B80 - 0x2CCF 14 个 erratum 检查 + reset
    Neoverse-N2 0x2CD0 - 0x2D?? 11 个 erratum 检查
    Neoverse-V1 0x2D?? - 0x2E?? 若干 erratum 检查
    AEM Generic 0x66E8 - .cpu_ops 通用 CPU 操作

    注意: 虽然 QEMU 平台运行时只使用一种核心，但 TF-A 静态链接了所有支持的 CPU errata 代码和 CPU ops 表，这显著增加了代码段体积。

    3.2.9 汇编辅助函数¶

    符号 地址 大小 说明
    el3_panic 0x2190 0x30 EL3 panic 处理
    smc_handler64 0x2084 0xBC SMC 异常路由（汇编）
    console_pl011_core_init 0x21C0 0x58 PL011 UART 初始化（汇编）
    flush_dcache_range 0x22A4 0x38 D-cache 刷新
    zeromem / zero_normalmem 0x22DC / 0x22E4 8 / 232 B 内存清零
    memcpy16 0x23CC 0x30 16 字节对齐拷贝
    disable_mmu_el3 0x23FC 0x1C 禁用 EL3 MMU
    reset_handler 0x2A30 0x20 CPU reset 处理
    get_cpu_ops_ptr 0x2A50 0x48 获取 CPU 操作表指针

    3.3.3 CPU Errata 表¶

    每种 CPU 核心都有对应的 .rodata.errata_entries 表，记录该核心的 erratum 编号和处理函数：

    CPU 核心 地址 大小
    Cortex-A53 0x57B8 0xD8 (216 B)
    Cortex-A55 0x5890 0xD8 (216 B)
    Cortex-A57 0x5968 0x180 (384 B)
    Cortex-A710 0x5AE8 0x1C8 (456 B)
    Cortex-A72 0x5CB0 0x78 (120 B)
    Cortex-A76 0x5D28 0x150 (336 B)
    Neoverse-N1 0x5E78 0x198 (408 B)
    Neoverse-N2 0x6010 0x1F8 (504 B)
    Neoverse-V1 0x6208 0x198 (408 B)
    AEM Generic 0x66E7 0x00 (空)
    QEMU Max 0x66E7 0x00 (空)

    总计约 3.3 KB 的 errata 表数据，占 .rodata 的近一半。

    3.3.4 镜像 UUID 规格¶

    BL1 通过 UUID 定位固件包中的各个镜像：

    符号 地址 说明
    rmm_uuid_spec 0x6646 RMM (Realm Management Monitor) UUID
    bl33_uuid_spec 0x6656 BL33 (Non-secure firmware, 如 U-Boot) UUID
    tos_fw_config_uuid_spec 0x6666 Trusted OS FW Config UUID
    tb_fw_config_uuid_spec 0x6676 Trusted Boot FW Config UUID
    bl32_extra2_uuid_spec 0x6686 BL32 Extra2 UUID
    bl32_extra1_uuid_spec 0x6696 BL32 Extra1 UUID
    bl32_uuid_spec 0x66A6 BL32 (Trusted OS, 如 OP-TEE) UUID
    bl31_uuid_spec 0x66B6 BL31 (EL3 Runtime firmware) UUID
    bl2_uuid_spec 0x66C6 BL2 UUID

    3.4 特殊只读 Section¶

    Section 地址 大小 说明
    .iplt 0x5000 0x00 间接 PLT（未使用，PIE 模式）
    .ARM.extab - 空 ARM 异常展开表
    .ARM.exidx - 空 ARM 异常展开索引

    4. ROM/RAM 运行时搬运机制¶

    BL1 采用了 ROM + RAM 分离布局：

    Flash/ROM 侧:                    RAM 侧:
┌──────────────┐                 ┌──────────────┐
│  .text       │                 │              │
│  .rodata     │     复制        │  .data       │ ← 从 ROM 搬运
│  .data(ROM)  │ ──────────────→ │  .stacks     │
│              │                 │  .bss        │ ← 运行时清零
└──────────────┘                 │  .xlat_table │ ← 运行时填充
                                 └──────────────┘
    • .data 段的 load address 在 ROM 中 (0x69F0)，VMA/RUN address 在 RAM 中 (0xE0EE000)
    • 启动时 bl1_entrypoint 汇编代码负责将 .data 从 ROM 搬运到 RAM
    • .bss 段在 RAM 中，需要清零
    • .stacks 段在 RAM 中，栈指针在入口处设置

    10. 关键分析结论¶

    10.1 空间效率¶

    1. ROM 使用率仅 20.9%，有大量空间裕量，这为未来添加功能（如安全启动增强、更多 IO 后端）提供了充足空间
    2. RAM 使用率 42.2%，.xlat_table 占用了 RAM 的 31.7%，是最大的单一数据段
    3. .text 段中 CPU errata 代码占比显著，对 QEMU 平台而言这些代码可能不会被运行，但增加了代码段体积

    10.2 启动流程推断¶

    从符号布局可以推断 BL1 的启动流程：

    bl1_entrypoint (汇编)          ← CPU 复位向量
  ├─ 禁用中断、设置 EL3 栈
  ├─ 搬运 .data 段 (ROM → RAM)
  ├─ 清零 .bss 段
  └─ 跳转 bl1_main (C)

bl1_main
  ├─ bl1_setup
  │    └─ bl1_early_platform_setup  (控制台、IO 初始化)
  │         └─ qemu_console_init
  │         └─ plat_qemu_io_setup
  │              └─ open_fip / open_memmap
  │
  ├─ bl1_plat_arch_setup
  │    └─ setup_page_tables         (配置初始页表)
  │    └─ enable_mmu_el3            (使能 MMU)
  │
  ├─ print_errata_status            (打印 CPU 勘误状态)
  ├─ NOTICE("BL1: ...")            (启动日志)
  │
  └─ bl1_prepare_next_image
       ├─ load_auth_image           (加载并认证 BL2)
       │    └─ plat_get_image_source
       │    └─ io_open / io_read
       │
       └─ cm_prepare_el3_exit       (准备跳转到 BL2)
            └─ ERET 到 BL2 入口

    10.3 安全相关观察¶

    1. BL1 支持镜像认证：load_auth_image 函数表明 BL1 可在加载 BL2 时进行签名验证
    2. EL3 执行：所有代码运行在最高特权级 EL3，smc_handler64 处理来自低 EL 的 SMC 调用
    3. 没有动态内存分配：所有数据结构都是静态分配（.bss / .data），没有 heap
    4. Bakery Lock 和 PMF Timestamp 区域大小为 0，表明 QEMU 单核场景下未启用多核同步和时间测量
    5. 无运行时服务描述符：__RT_SVC_DESCS__ 区域为空，BL1 不注册任何 runtime service
    6. 无动态重定位：.rela.dyn 和 .got 均为空，BL1 完全静态链接，无 PIE 依赖
    7. 完全静态链接：没有 .iplt/.igot.plt 条目，所有符号引用在链接时已解析

    10.4 Cross Reference Table¶

    本次 map 文件不包含 Cross Reference Table（符号交叉引用表）。这是因为链接器未使用 --cref 选项。如需分析函数间的调用关系，可通过以下方式补充：

    • 在 TF-A 构建系统中向 LDFLAGS 添加 -Map=bl1.map --cref 重新链接
    • 使用 readelf -s bl1.elf 获取符号表，结合源码静态分析调用链
    • 使用 GDB 的 info functions 和 info callers 命令动态追踪

    10.4 优化建议（供参考）¶

    1. CPU errata 代码裁剪：对于 QEMU 平台，可通过 Kconfig ERRATA_A53_xxx 等选项精确启用所需的 errata，减少不必要的代码
    2. CPU ops 裁剪：11 种 CPU ops 表（768 B）中，QEMU 只需 1-2 种
    3. .xlat_table 优化：24 KB 页表对 QEMU 简单地址映射可能过大，可考虑减少预留的页表数量
