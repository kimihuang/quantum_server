    TF-A v2.12 BL1 启动流程与关键功能分析¶

    1. BL1 概述¶

    1.1 角色定位¶

    BL1 (Boot Loader Stage 1) 是 TF-A 启动链的第一个阶段，运行在 EL3（最高特权级），负责：

    1. CPU/架构级早期初始化（MMU、缓存、异常向量）
    2. 平台硬件初始化（控制台、IO 子系统）
    3. 从存储介质加载 BL2 镜像（支持认证）
    4. 准备执行上下文并跳转到 BL2
    ┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐     ┌──────────┐
│  BL1    │────→│  BL2    │────→│  BL31   │────→│ BL32/33 │────→│  Kernel  │
│ (EL3)   │     │ (EL3)   │     │ (EL3)   │     │(EL1/EL2)│     │ (EL1/2)  │
│ ROM/RAM │     │ Trusted │     │ Runtime │     │ OS/fw   │     │          │
└─────────┘     └─────────┘     └─────────┘     └─────────┘     └──────────┘
   QEMU: ROM=0x0
         RAM=0xE0EE000

    1.2 内存布局（map 文件验证）¶

    区域 地址范围 大小 说明
    .text 0x0000 - 0x4FFF 20 KB 代码段（含向量表）
    .rodata 0x5000 - 0x69E7 6.6 KB 只读数据
    .data ROM 副本 0x69F0 - 0x6AF4 261 B 运行时搬运到 RAM
    .data (RAM) 0xE0EE000 - 0xE0EE104 261 B 初始化数据
    .stacks (RAM) 0xE0EE140 - 0xE0EF13F 4 KB 运行时栈
    .bss (RAM) 0xE0EF140 - 0xE0EF9DF 2.2 KB 零初始化数据
    .xlat_table (RAM) 0xE0F0000 - 0xE0F5FFF 24 KB MMU 页表

    3. Phase 1: 汇编入口点详解 (bl1_entrypoint)¶

    源文件: bl1/aarch64/bl1_entrypoint.S 符号地址: 0x0000 (map 文件) 大小: 0x100 (256 B)

    3.1 el3_entrypoint_common 宏展开¶

    BL1 入口点通过 el3_entrypoint_common 宏实现（定义在 include/arch/aarch64/el3_common_macros.S），该宏由以下参数配置：

    el3_entrypoint_common                  \
    _init_sctlr=1                      \  ← 初始化 SCTLR_EL3
    _warm_boot_mailbox=!PROGRAMMABLE_RESET_ADDRESS \  ← QEMU 使用可编程复位地址，跳过
    _secondary_cold_boot=!COLD_BOOT_SINGLE_CPU \  ← 支持从核冷启动
    _init_memory=1                     \  ← 使能 MMU、搬运 .data、清零 .bss
    _init_c_runtime=1                  \  ← C 运行时初始化
    _exception_vectors=bl1_exceptions  \  ← 异常向量表基地址
    _pie_fixup_size=0                     ← 无 PIE 修复

    3.1.1 el3_arch_init_common — 架构寄存器初始化¶

    ┌─────────────────────────────────────────────────┐
│ SCTLR_EL3:                                      │
│   M=0 (MMU 关闭)                                │
│   I=0,C=0 (指令/数据缓存关闭)                    │
│   A=0 (对齐检查关闭)                             │
│   SA=0,EE=0 (小端序)                            │
│   WXN=0 (不允许 Writable-X)                     │
│                                                  │
│ SCR_EL3:                                         │
│   NS=1 (默认 Non-Secure)                        │
│   IRQ/FIQ=0 (路由到 EL3)                        │
│   EA=1 (External Abort 路由到 EL3)              │
│   SIF=1 (Secure FIQ 路由到 EL3)                 │
│   RW=0 (EL3 为 Secure)                          │
│                                                  │
│ MDCR_EL3: SPME=0 (禁止 EL2 PMU 访问)           │
│ CPTR_EL3: 禁止 EL2/EL1 SIMD/SVE/FP 访问         │
└─────────────────────────────────────────────────┘

    3.1.2 Warm Boot 检查¶

    由于 QEMU 的复位地址是可编程的（PROGRAMMABLE_RESET_ADDRESS=1），_warm_boot_mailbox=!PROGRAMMABLE_RESET_ADDRESS 使得 warm boot 路径被跳过。每次都走冷启动路径。

    3.1.3 VBAR_EL3 设置¶

    adr  x0, bl1_exceptions      /* x0 = 向量表物理地址 0x4000 */
msr  VBAR_EL3, x0           /* 设置异常向量表基址寄存器 */

    3.1.4 CPU Reset 处理¶

    bl  reset_handler            /* 调用 cpu_helpers.o 中的 reset_handler */

    reset_handler 的核心逻辑： 1. 调用 get_cpu_ops_ptr → 通过 .cpu_ops 表获取当前 CPU 的操作函数集 2. 调用 cpu_ops->reset_func → 执行 CPU 特定的 reset 函数 - QEMU 平台匹配到 qemu_max_ops 或 aem_generic_ops - 每个 CPU 的 reset_func 内部会遍历该 CPU 的 errata 表，逐个检查并应用 workaround 3. 调用 cpu_ops->errata_func → 打印 errata 状态

    3.1.5 冷启动路径：主/从核分发¶

    bl  plat_set_my_stack         /* 设置 EL3 栈指针 = platform_up_stack.o 中的栈顶 */
bl  plat_is_my_cpu_primary    /* 读取 MPIDR，判断是否主 CPU */
    ├─ 主核: 继续执行
    └─ 从核:
         bl  plat_secondary_cold_boot_setup
              └─ QEMU: 空函数 (4 bytes, 0x3104)
              实际场景: 等待主核通过 SEV 指令唤醒
         b   .     ← 死循环等待 (PSCI cpu_on 触发后唤醒)

    map 验证: plat_set_my_stack 在 0x30C8 (16 B)，plat_is_my_cpu_primary 在 0x30F0 (20 B)，plat_secondary_cold_boot_setup 在 0x3104 (52 B)

    3.1.6 内存初始化 (_init_memory=1)¶

    ┌─────────────────────────────────────────────────────┐
│  1. zeromem .bss 段                                  │
│     └─ 调用 zeromem (0x22DC) / zero_normalmem (0x22E4)│
│     └─ 将 __BSS_START__ 到 __BSS_END__ 清零          │
│                                                      │
│  2. 搬运 .data 段 (ROM → RAM)                        │
│     └─ 从 __DATA_ROM_START__ (0x69F0)                │
│     └─ 到 __DATA_RAM_START__ (0xE0EE000)             │
│     └─ 大小: __DATA_SIZE__ = 0x105 (261 bytes)       │
│                                                      │
│  3. 使能 MMU                                         │
│     └─ enable_mmu_direct_el3 (0x305C, 88 B)          │
│     └─ 设置 MAIR_EL3, TCR_EL3, TTBR0_EL3              │
│     └─ 使能 SCTLR_EL3.M (MMU 位)                     │
│     └─ ISB 同步                                       │
│     └─ TLB 无效化 (TLBI ALLE3)                        │
└─────────────────────────────────────────────────────┘

    关键: MMU 使能前，页表已在 bl1_plat_arch_setup() 中由 setup_page_tables() 预先填充。但此处 enable_mmu_direct_el3 使用的是"直通"映射方式，在 bl1_plat_arch_setup() 中的 enable_mmu_el3() 才会启用完整的虚实地址映射。

    实际上，el3_entrypoint_common 中的内存初始化是在 bl1_setup() 调用之前执行的。对于 BL1，.data 搬运和 MMU 使能发生在 bl1_setup() 返回之后，由 bl1_plat_arch_setup() 中的 enable_mmu_el3(0) 完成。el3_entrypoint_common 的 _init_memory=1 主要负责 .bss 清零。

    3.1.7 C 运行时初始化 (_init_c_runtime=1)¶

    ┌─────────────────────────────────────────────────────┐
│  1. 再次清零 .bss 段 (二次确认)                      │
│                                                      │
│  2. 初始化 SP (栈指针已由 plat_set_my_stack 设置)     │
│                                                      │
│  3. 初始化 GOT 表 (本次: 空, 静态链接)                │
│                                                      │
│  4. 初始化 BSS 段中指向 data 段的指针                 │
│     └─ 将 .data 段中的全局变量地址写入 GOT             │
│     └─ 本次 GOT 为空, 跳过                            │
│                                                      │
│  5. 调用 plat_report_exception → QEMU: 空函数          │
│                                                      │
│  6. 跳转到 C 入口: bl bl1_setup                      │
└─────────────────────────────────────────────────────┘

    3.2 Phase 1 函数调用总结¶

    bl1_entrypoint (0x0000)

  │    ├─ 写 MDCR_EL3

  ├─ zeromem (.bss) (0x22DC)

    4. Phase 2: bl1_setup() — 早期平台初始化¶

    源文件: bl1/bl1_main.c:45 符号地址: 0x01D4 (map 文件) 大小: 0x14 (20 B)

    void bl1_setup(void)
{
    plat_setup_early_console();          // EARLY_CONSOLE=0 时为空
    bl1_early_platform_setup();          // 平台早期设置
    bl1_plat_arch_setup();              // 平台架构设置 (MMU)
}

    4.1 bl1_early_platform_setup() — QEMU 平台早期设置¶

    源文件: plat/qemu/common/qemu_bl1_setup.c:52

    void bl1_early_platform_setup(void)
{
    qemu_console_init();                // 初始化 PL011 UART
    bl1_tzram_layout.total_base = BL_RAM_BASE;   // 0xE0EE000
    bl1_tzram_layout.total_size = BL_RAM_SIZE;   // RAM 大小
}

    qemu_console_init() 调用链¶

    qemu_console_init (qemu_console.o:0x1E50)
  └─ console_pl011_register(PL011_UART0_BASE, ...)
       ├─ console_pl011_core_init → 配置 PL011 寄存器
       │   (0x21C0, 88 B)
       ├─ console_register → 注册到多控制台框架
       │   (0x0580, 52 B)

    BL1 的 MMU 在 bl1_plat_arch_setup() 中使能，使用三级页表（4KB granule）：

    enable_mmu_el3 流程:
  1. setup_mmu_cfg:
     ├─ MAIR_EL3 = Attr0(Device-nGnRnE) | Attr1(Normal-WriteBack)
     ├─ TCR_EL3:
     │   ├─ T0SZ = 地址大小 (如 40 bit)
     │   ├─ TG0 = TG0_4K (4KB granule)
     │   ├─ SH0 = INNER (Inner Shareable)
     │   ├─ ORGN0GN = WBWA (Write-Back, Write-Allocate)
     │   └─ TBI0 = 1 (Top Byte Ignored)
     └─ TTBR0_EL3 = 页表基地址

  2. init_xlat_tables → 遍历 mmap 填充页表
     └─ .xlat_table (0xE0F0000, 24 KB) = 6 个 L0 页表项范围

  3. 使能:
     SCTLR_EL3.M = 1, .C = 1, .I = 1
     TLBI ALLE3 → 无效化 TLB

    BL1 使用 cpu_context_t 结构保存完整的 CPU 上下文（约 1152 B）：

       │
       ├─ cpu_context: 整体清零
       │
       ├─ el3_sysregs_context:
       │   ├─ scr_el3    (安全状态, 中断路由)
       │   ├─ sctlr_el3  (EL3 系统控制)
       │   ├─ cptr_el3   (陷阱控制)
       │   ├─ mdcr_el3   (调试控制)
       │   ├─ elr_el3    = BL2 PC (返回地址)
       │   ├─ spsr_el3   = BL2 SPSR (CPSR)
       │   └─ arguments (args[0..7])
       │
       ├─ el1_sysregs_context (BL2 在 EL1/EL2 使用):
       │   ├─ sctlr_el1, actlr_el1, cpacr_el1
       │   └─ tcr_el1, ttbr0_el1, ttbr1_el1, mair_el1
       │
       └─ el2_sysregs_context (BL2 在 EL2 时):
           ├─ hcr_el2, vtcr_el2, sctlr_el2

    7.5 安全启动链 (Trusted Boot)¶

    当 TRUSTED_BOARD_BOOT=1 时，BL1 的镜像加载流程变为：

    load_auth_image_recursive(image_id)
  ├─ auth_mod_get_parent_id(image_id) → parent_id
  ├─ 递归: load_auth_image_recursive(parent_id)
  │   └─ 直到 root of trust (trusted key cert)
  ├─ load_image(image_id)
  │   └─ 从 FIP/Semihosting 加载二进制
  └─ auth_mod_verify_img(image_id, base, size)
      ├─ 使用 RSA/ECDSA 验证签名
      ├─ 验证证书链
      └─ 失败时: zero_normalmem + flush_dcache_range → 清除敏感数据

    7.6 SMC 中断处理¶

    BL1 的 SMC handler (bl1_smc_handler, 0x0318, 100 B) 在 BL1 运行期间处理来自低 EL 的 SMC 调用：

    bl1_smc_handler (bl1_main.c:228)
  ├─ BL1_SMC_CALL_COUNT → 返回支持的 SMC 数量
  ├─ BL1_SMC_UID → 返回 BL1 服务 UUID
  ├─ BL1_SMC_VERSION → 返回版本号
  └─ TRUSTED_BOARD_BOOT 时:

    SMC 异常由 smc_handler64 (bl1_exceptions.o:0x2084, 188 B) 接收并路由到 bl1_smc_handler。

    8. 从核启动路径¶

    QEMU 平台支持多核启动，从核的路径：

    CPU n 复位
  └─ bl1_entrypoint
       └─ el3_entrypoint_common
            ├─ el3_arch_init_common (架构初始化)
            ├─ VBAR_EL3 = bl1_exceptions
            ├─ reset_handler → errata workaround
            ├─ plat_set_my_stack (设置从核栈)
            └─ plat_is_my_cpu_primary → FALSE
                 └─ plat_secondary_cold_boot_setup
                      └─ QEMU: 空函数 (PSCI cpu_on 触发)
                 └─ b .  ← 死循环等待

... 主核通过 PSCI CPU_ON 通知 ...

CPU n 被唤醒 (Warm Boot)
  └─ bl1_entrypoint (warm boot)
       └─ el3_entrypoint_common
            └─ warm_boot_mailbox 机制
                 └─ 读取 mailbox 中的入口地址
                 └─ 跳转到主核指定的入口 (通常为 BL31)

    struct plat_io_policy {
    uintptr_t *dev_handle;            // IO 设备句柄指针
    uintptr_t image_spec;             // 镜像定位规格 (UUID/地址/文件名)
    int (*check)(const uintptr_t);     // 检查函数 (open_fip/open_memmap)
};

    9.3 cpu_context_t — CPU 上下文¶

    bl1_cpu_context[2] (BSS: 0xE0EF140, 1152 B) 包含完整的 CPU 寄存器状态，分为： - el3_sysregs_context: EL3 寄存器（SCR, SCTLR, CPTR, MDCR, ELR, SPSR 等） - el1_sysregs_context: EL1 寄存器（SCTLR, TCR, TTBR, MAIR, CPACR 等） - el2_sysregs_context: EL2 寄存器（HCR, VTCR, SCTLR, CPTR 等）
