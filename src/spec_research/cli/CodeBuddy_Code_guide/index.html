    CodeBuddy Code 操作文档¶

    CodeBuddy Code 是腾讯云推出的 AI 驱动终端编程助手，提供从代码编写到项目部署的全生命周期 AI 辅助开发能力。

    1. 环境要求与安装¶

    环境要求¶

    • Node.js: 18.20 或更高版本
    • 操作系统: macOS / Linux / Windows

    安装方式¶

    方式一：npm 全局安装（推荐）

    npm install -g @tencent-ai/codebuddy-code

    方式二：原生安装脚本（无需 Node.js）

    # macOS / Linux
curl -fsSL https://copilot.tencent.com/cli/install.sh | bash

# Windows (PowerShell)
irm https://copilot.tencent.com/cli/install.ps1 | iex

    验证安装¶

    codebuddy --version
# 或使用别名
cbc --version

    cbc 是 codebuddy 的简写别名。

    更新¶

    codebuddy update

    2. 首次启动与认证¶

    启动¶

    cd /path/to/your/project
codebuddy

    首次启动时会弹出登录方式选择：

    登录方式 适用场景
    中国站 中国大陆用户
    国际站 海外用户
    企业域名 私有部署，需要自定义服务器地址
    iOA 腾讯内部员工

    使用 ↑↓ 选择，Enter 确认，会打开浏览器完成认证。

    初始化项目上下文（强烈推荐）¶

    > /init

    /init 的作用：

    • 让 CodeBuddy 更准确地理解代码结构和业务逻辑
    • 加快响应速度（避免重复文件扫描）
    • 减少 30-50% 的 Token 消耗
    • 避免提出不兼容的修改方案

    设置响应语言¶

    > /config

    在配置面板中设置 Language 选项，例如选择"简体中文"。

    3. 交互模式使用¶

    基本操作¶

    进入交互模式后，直接输入自然语言即可与 CodeBuddy 对话：

    > 帮我分析这个项目的结构
> 为这个 API 接口写单元测试
> 这段代码有什么潜在的性能问题？

    键盘快捷键¶

    导航与输入¶

    快捷键 功能
    ↑ / ↓ 浏览命令历史
    Tab 自动补全 / 切换思考模式
    Enter 发送消息
    Shift+Enter 换行（需先运行 /terminal-setup）
    \ + Enter 换行（通用方式）
    Ctrl+J 插入换行符
    Ctrl+C 取消当前输入或生成
    Ctrl+D 退出（输入为空时按两次）
    Ctrl+L 清屏（对话历史保留）
    Esc + Esc 回滚对话/代码到之前状态
    Ctrl+G 在外部编辑器中编辑提示词

    权限与模式¶

    快捷键 功能
    Shift+Tab (macOS/Linux) 循环切换权限模式
    Alt+M (Windows) 循环切换权限模式
    Alt+P (macOS) / Alt+P (Windows/Linux) 切换模型

    面板与输出¶

    快捷键 功能
    Ctrl+R 反向搜索命令历史
    Ctrl+O 展开/折叠详细输出
    Ctrl+Y 复制最近一条 AI 回复到剪贴板
    Ctrl+V (macOS/Linux) / Alt+V (Windows) 粘贴（支持文本、图片和图片文件路径）
    Ctrl+B 将当前 Bash 命令移至后台执行

    输入前缀¶

    前缀 功能 示例
    / 斜杠命令 /help
    ! Bash 模式 - 直接运行命令 ! npm test
    @ 文件路径引用 @src/index.ts
    # 记忆快捷方式 - 添加内容到 CODEBUDDY.md # 始终使用 TypeScript

    多行输入方式¶

    方式 快捷键 说明
    反斜杠换行 \ + Enter 所有终端通用
    Shift+Enter Shift+Enter 运行 /terminal-setup 后可用
    控制序列 Ctrl+J 插入换行符
    直接粘贴 粘贴多行文本 自动识别为多行输入

    Bash 模式¶

    在输入前加 ! 可直接运行 Shell 命令，无需 AI 解释或权限审批：

    ! npm test
! git status
! ls -la

    命令及输出会被添加到对话上下文中。

    Vim 编辑模式¶

    通过 /vim 命令启用，支持：

    • 模式切换: Esc 进入 NORMAL 模式，i/a/o 进入 INSERT 模式
    • 导航: h/j/k/l 移动，w/b 跳词，0/$ 行首行尾
    • 编辑: x 删字符，dd 删行，cc 改行，. 重复上次修改

    4. CLI 命令参考¶

    基础命令¶

    codebuddy                    # 启动交互式 REPL
codebuddy "帮我优化这个函数"   # 带初始提示词启动
cbc "查询"                   # 使用别名

    非交互模式（One-Shot）¶

    # 直接查询后退出
codebuddy -p "优化这个 SQL 查询的性能"

# 管道输入
cat error.log | codebuddy -p "分析这些错误日志"

# 需要文件访问时必须加 -y
codebuddy -p "审查 src/utils.js 的代码质量" -y

# 指定输出格式
codebuddy -p "查询" --output-format json

# 限制代理轮次数
codebuddy -p "查询" --max-turns 3

    根据 Prompt 修改指定文件¶

    使用 -p 单次命令模式可以直接通过 prompt 修改文件，关键在于正确使用 -y 参数和权限模式。

    基本语法：

    codebuddy -p "修改指令" -y

    核心要点：只要 prompt 意图涉及文件读写或命令执行，必须添加 -y（--dangerously-skip-permissions）参数，否则单次命令模式无法获得文件修改权限。

    ⚠️ 安全警告：-y 会跳过所有权限确认，AI 可不经审批执行任意命令和文件操作。仅在 CI/CD、沙箱等隔离可信环境中使用，生产代码仓库中请优先使用 --permission-mode acceptEdits 或 --permission-mode plan 替代。

    修改指定文件示例：

    # 修改单个文件
codebuddy -p "将 src/utils.js 中的 formatDate 函数改为支持时区参数" -y

# 修改多个文件
codebuddy -p "把 src/a.ts 和 src/b.ts 中的接口定义统一到 src/types.ts" -y

# 批量修改
codebuddy -p "将所有组件从 class 组件迁移到函数组件" -y
codebuddy -p "为 services 目录下的所有文件添加单元测试" -y

# 项目级代码检查与修改
codebuddy -p "检查整个项目的 TypeScript 类型定义并修复错误" -y

    权限参数选择：

    参数 说明 适用场景
    -y 跳过所有权限确认 CI/CD、沙箱环境
    --permission-mode acceptEdits 自动批准文件编辑，其他操作仍需确认 日常自动化
    --permission-mode plan 只生成修改计划，审批后再执行 需要审查的修改
    --permission-mode bypassPermissions 跳过所有权限检查（等同于 -y） 隔离沙箱环境

    -y 参数使用判断：

    操作类型 需要 -y? 示例
    纯文本问答 不需要 codebuddy -p "解释这段代码"
    管道输入分析 不需要 cat log \| codebuddy -p "分析"
    读取/修改文件 必须 codebuddy -p "修改文件" -y
    执行命令 必须 codebuddy -p "运行测试" -y

    交互模式中使用 @ 指定文件：

    在交互模式中，输入 @ 可触发文件路径自动补全，精准指定要修改的文件：

    > 把 @src/utils.js 里的 log 都改成用 debug 库
> @src/config.ts 和 @src/database.ts 的连接逻辑有重复，帮我合并

    会话管理¶

    codebuddy -c                          # 继续最近的对话
codebuddy -c -p "继续我的任务"          # SDK 模式继续对话
codebuddy --resume                    # 交互式选择会话恢复
codebuddy --resume abc123 "继续工作"    # 按 ID 恢复会话

    后台运行¶

    codebuddy --bg "实现登录页面"                  # 后台运行
codebuddy --bg --name feature-x "实现功能"     # 命名后台会话
codebuddy ps                                  # 列出活跃进程
codebuddy logs feature-x                      # 查看日志
codebuddy attach feature-x                    # 附加到后台会话
codebuddy kill feature-x                      # 终止后台会话

    Daemon 守护进程¶

    codebuddy daemon start [--port 8080]  # 启动
codebuddy daemon stop                 # 停止
codebuddy daemon status               # 状态查看
codebuddy daemon restart              # 重启

    常用 CLI 参数¶

    参数 说明
    -p / --print 打印响应后退出（非交互模式）
    -y 跳过权限提示（⚠️ 跳过所有权限确认，仅在可信环境中使用）
    --model <model> 指定模型
    --agent <name> 指定 agent
    --add-dir <path> 添加额外工作目录
    --plugin-dir <path> 加载本地插件
    --permission-mode <mode> 设置权限模式
    --serve 启动 HTTP 服务
    --worktree [name] 在 git worktree 中运行
    --sandbox 在沙箱中运行
    --verbose 详细日志
    --debug 调试模式

    管道与 Unix 集成¶

    # 管道输入分析（注意：管道输入必须加 -p 参数）
git log --oneline | codebuddy -p "分析这些提交，找出可能的问题"
cat error.log | codebuddy -p "帮我分析这些错误日志"

# 输出到文件
cat build-error.txt | codebuddy -p '解释构建错误的根本原因' > output.txt

# 在 CI/CD 中使用
codebuddy -p '检查代码变更中的拼写错误' -y

    6. 常见工作流¶

    理解新代码库¶

    > 给我这个代码库的概览
> 解释这里使用的主要架构模式
> 关键的数据模型有哪些？
> 找到处理用户身份验证的文件

    修复 Bug¶

    > 运行 npm test 时看到错误 <粘贴错误信息>
> 建议几种修复方案
> 更新 user.ts 添加你建议的空值检查

    代码重构¶

    > 在代码库中查找已弃用的 API 使用
> 建议如何重构 utils.js 以使用现代 JavaScript 特性
> 重构 utils.js，保持相同的行为
> 为重构的代码运行测试

    编写测试¶

    > 查找未被测试覆盖的函数
> 为通知服务添加测试
> 为通知服务添加边缘条件的测试用例
> 运行新测试并修复任何失败

    创建 Merge Request¶

    > 总结我最近的代码更改
> 创建一个 MR
> 用更多上下文增强 MR 描述
> 添加关于如何测试这些更改的信息

    使用计划模式¶

    按 Shift+Tab（macOS/Linux）或 Alt+M（Windows）切换到计划模式：

    > 我需要重构身份验证系统以使用 OAuth2。创建详细的迁移计划。
> 向后兼容性怎么办？
> 我们应该如何处理数据库迁移？

    计划确认后再执行，适合多步骤实现和安全分析。

    引用文件¶

    > 解释 @src/utils/auth.js 中的逻辑        # 引用单个文件
> @src/components 的结构是什么？          # 引用目录
> @file1.js 和 @file2.js 的区别是什么？    # 引用多个文件

    处理图片¶

    • 拖放：将图片拖入 CodeBuddy 窗口
    • 粘贴：Ctrl+V 粘贴
    • 路径引用：> 分析这张图片：/path/to/image.png

    扩展思考¶

    在提示词中加入"思考"或"深入思考"关键词启用深度推理：

    > 深入思考实现 OAuth2 认证系统的最佳方法
> 思考这种方法的潜在安全漏洞

    7. 配置系统¶

    配置文件层级¶

    层级 文件路径 用途 Git 跟踪
    用户级 ~/.codebuddy/settings.json 全局个人偏好 否
    项目共享 .codebuddy/settings.json 团队共享的项目标准 是
    项目本地 .codebuddy/settings.local.json 个人的项目覆盖 否

    四种权限模式¶

    模式 说明 CLI 参数
    Normal（默认） 根据权限规则逐个确认 -
    Auto-accept Edits 自动批准文件编辑，其他仍需确认 --permission-mode acceptEdits

    切换方式¶

    权限规则¶

    {
  "permissions": {
    "allow": ["Bash(npm run test:*)", "Read"],

    "deny": ["WebFetch", "Bash(curl:*)", "Read(./.env)"],

    11. Skills 技能系统¶

    Skills 是 AI 根据任务需求自动选择的专业能力模块。

    与斜杠命令的区别¶

    特性 Skills 斜杠命令
    触发方式 AI 自动识别调用 用户手动输入
    使用场景 专业领域任务 快捷操作
    可见性 对用户透明 用户主动发起

    创建 Skill¶

    在 .codebuddy/skills/<skill-name>/SKILL.md（项目级）或 ~/.codebuddy/skills/<skill-name>/SKILL.md（用户级）创建文件：

    ---
name: pdf
description: PDF 文档处理专家。当用户需要处理 PDF 相关任务时使用。
allowed-tools: Read, Write, Bash
---
你是一个 PDF 文档处理专家。

## 核心能力
- 提取 PDF 文本内容
- 转换 PDF 格式
- 合并和拆分 PDF

## 工作流程
1. 检查 PDF 文件是否存在
2. 使用适当工具提取内容
3. 根据需求处理

    Frontmatter 字段¶

    字段 说明
    name Skill 名称（默认使用目录名）
    description 描述（帮助 AI 决定何时使用）
    allowed-tools 允许使用的工具白名单
    disable-model-invocation 设为 true 时仅支持手动调用
    context 设为 fork 时在独立子代理中执行
    agent 指定子代理类型（配合 context: fork）

    支持的语法¶

    • $ARGUMENTS：参数占位符
    • @file：文件引用
    • !`command`：Shell 命令执行
    • ${CODEBUDDY_SKILL_DIR}：Skill 目录路径
    • ${ENV_VAR}：环境变量

    查看 Skills¶

    /skills

    CLI 命令方式使用 Skills¶

    除了交互模式下的自动识别和手动调用，Skills 也可以在 CLI 命令行和非交互模式下使用。

    方式一：在 -p 模式中通过提示词触发 Skill¶

    AI 模型会根据 prompt 内容与 Skill 的 description 字段匹配度自动调用 Skill，行为与交互模式一致：

    # AI 根据 prompt 语义自动匹配 Skill
codebuddy -p "帮我处理 report.pdf，提取其中的文本内容" -y

# 明确指定使用某个 Skill
codebuddy -p "使用 pdf skill 将 report.pdf 转为 markdown" -y

# 带参数传递（通过 prompt 描述参数）
codebuddy -p "用 code-review skill 审查 src/auth.ts 和 src/user.ts" -y

    关键：Skill 的 description 字段决定了 AI 是否能正确匹配。编写清晰、包含关键词的 description 是 CLI 模式下可靠触发 Skill 的前提。

    方式二：通过 /skill-name 语法在命令行触发¶

    # 使用斜杠命令语法触发 Skill（等同于交互模式下的手动调用）
codebuddy -p "/pdf 提取 report.pdf 的文本" -y
codebuddy -p "/code-review src/utils.ts" -y
codebuddy -p "/deep-research React 性能优化方案" -y

    注意：此方式对 disable-model-invocation: false（默认）且 user-invocable: true（默认）的 Skill 有效。

    方式三：在脚本中调用 Skill¶

    #!/bin/bash
# 使用 Skill 自动审查 MR
gh pr diff 123 | codebuddy -p \
  "使用 code-review skill 审查此 diff，重点关注安全问题" \
  --output-format json \
  -y \
  --allowedTools "Skill(code-review),Read,Grep" \
  | jq -r '.result'

# 使用 Skill 处理文档
codebuddy -p "/pdf convert /data/report.pdf to markdown" \
  --output-format json \
  -y \
  | jq -r '.result'

    工具权限中的 Skill 控制¶

    在 --allowedTools 和 --disallowedTools 中可以使用 Skill 工具名来精确控制：

    # 仅允许使用指定的 Skill
codebuddy -p "审查代码" -y --allowedTools "Skill(code-review),Read,Grep"

# 禁止使用特定 Skill
codebuddy -p "分析项目" -y --disallowedTools "Skill(pdf)"

# 允许所有 Skill
codebuddy -p "处理文件" -y --allowedTools "Skill"

    在配置文件中管理 Skill 权限¶

    {
  "permissions": {
    "allow": ["Skill(code-review)", "Skill(pdf)"],
    "deny": ["Skill(internal-tool)"]
  }
}

    CLI 模式下 Skill 使用注意事项¶

    要点 说明
    自动匹配依赖 description CLI 模式下没有对话历史辅助，Skill 的 description 必须包含足够的关键词
    -y 参数必需 CLI 非交互模式下执行 Skill 需要跳过权限确认
    disable-model-invocation: true 的 Skill 只能通过 /skill-name 语法手动触发，AI 不会自动匹配
    user-invocable: false 的 Skill 对用户不可见，仅供 AI 内部调用
    项目级 Skill 优先 项目级 Skills（.codebuddy/skills/）优先于用户级（~/.codebuddy/skills/）
    插件 Skill 自动加载 通过插件安装的 Skill 会自动注册，无需额外配置

    13. Hooks 钩子系统¶

    Hooks 是在 CodeBuddy 生命周期各阶段自动执行的 Shell 命令。

    事件类型¶

    事件 触发时机 用途
    PreToolUse 工具调用前（可阻止执行） 权限控制、验证、日志
    PostToolUse 工具调用后 自动格式化、后处理
    UserPromptSubmit 用户提交提示词后 输入验证/修改
    Notification 发送通知时 自定义提醒
    Stop 完成响应时 清理、总结
    SubagentStop 子代理完成时 子任务跟踪
    SessionStart 会话开始时 环境设置
    SessionEnd 会话结束时 清理、日志

    配置示例¶

    在 settings.json 中配置：

    {
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | { read f; if echo \"$f\" | grep -q '\\.ts$'; then npx prettier --write \"$f\"; fi; }"
          }
        ]
      }
    ]
  }
}

    交互式设置¶

    /hooks

    选择事件 -> 添加 matcher -> 添加 hook 命令 -> 选择存储位置。

    安全提示：Hooks 使用当前用户环境凭证自动运行，仅注册可信的 Hook。

    MCP（Model Context Protocol）用于集成第三方工具和服务。

    配置文件位置¶

    传输类型¶

    STDIO（本地进程）

      "command": "python",

    HTTP（远程服务）

    {
  "mcpServers": {

      "type": "http",

  }
}

    SSE（Server-Sent Events）

    {
  "mcpServers": {
    "sse-server": {
      "type": "sse",

    }
  }
}

    17. 脚本与自动化集成¶

    无头模式（Headless Mode）是 CodeBuddy 在脚本和 CI/CD 中使用的核心方式，通过 -p 参数以非交互模式运行并返回结果。

    关键命令参数¶

    参数 说明 示例
    -p / --print 非交互模式，打印结果后退出 codebuddy -p "查询"
    -y 跳过权限确认（脚本中必需） codebuddy -p "任务" -y
    --output-format 输出格式：text / json / stream-json --output-format json
    --input-format 输入格式（配合 stream-json） --input-format stream-json
    --append-system-prompt 追加系统提示词 --append-system-prompt "角色"
    --allowedTools 允许的工具列表（空格或逗号分隔） --allowedTools "Bash,Read,Grep"
    --disallowedTools 禁止的工具列表 --disallowedTools "Bash(git push)"
    --json-schema 指定结构化 JSON 输出的 Schema 见下方示例
    --settings 从 JSON 文件或字符串加载设置 --settings '{"model":"gpt-5"}'
    --mcp-config 从 JSON 文件加载 MCP 服务器配置 --mcp-config servers.json
    --setting-sources 指定配置源（user/project/local） --setting-sources project,local

    重要：脚本和 CI 环境中没有交互终端，必须使用 -y 参数，否则所有需要授权的操作（文件读写、命令执行等）都会被阻止。

    输入输出格式¶

    文本输入输出（默认）¶

    # 直接传参
codebuddy -p "解释这段代码"

# 管道传入
echo "解释这段代码" | codebuddy -p

# 管道传入外部数据作为上下文
gh pr diff 123 | codebuddy -p "审查此 diff 的安全问题" -y

    JSON 输出¶

    # 获取包含元数据的结构化响应
codebuddy -p "总结这个项目" --output-format json

# 提取文本结果
codebuddy -p "总结这个项目" --output-format json | jq -r '.result'

    结构化 JSON 输出（配合 --json-schema）¶

    codebuddy -p "提取 auth.py 中的主要函数名称" \
  --output-format json \
  --json-schema '{"type":"object","properties":{"functions":{"type":"array","items":{"type":"string"}}},"required":["functions"]}' \
  | jq '.structured_output'

    流式 JSON 输入输出（stream-json）¶

    通过 stdin 以 jsonl 格式逐行发送消息，适合多轮对话脚本：

    # 单条消息
echo '{"type":"user","message":{"role":"user","content":[{"type":"text","text":"分析日志"}]}}' | \
  codebuddy -p --input-format stream-json --output-format stream-json

# 多轮对话（持续向同一进程发送）
printf '%s\n' \
  '{"type":"user","message":{"role":"user","content":[{"type":"text","text":"第一问"}]}}' \
  '{"type":"user","message":{"role":"user","content":[{"type":"text","text":"第二问"}]}}' \
  | codebuddy -p --input-format stream-json --output-format stream-json

    Shell 脚本示例¶

    示例 1：自动化安全审查¶

    #!/bin/bash
# security-review.sh

pr_number="${1:?用法: $0 <PR号>}"

gh pr diff "$pr_number" | codebuddy -p \
  --append-system-prompt "你是一名安全工程师。审查此 PR 的漏洞、不安全模式和合规问题。" \
  --output-format json \
  -y \
  --allowedTools "Read,Grep,WebSearch" \
  | jq -r '.result'

    示例 2：带错误处理和超时控制¶

    #!/bin/bash
# auto-fix.sh

prompt="${1:?用法: $0 <prompt>}"

if ! timeout 300 codebuddy -p "$prompt" \
  --output-format json \
  -y \
  --allowedTools "Bash,Read,Grep,Edit" \
  2>error.log; then
  echo "错误或超时:" >&2
  cat error.log >&2
  exit 1
fi

    示例 3：多轮对话（带会话持久性）¶

    #!/bin/bash
# multi-turn.sh

codebuddy -p --resume "$session_id" "检查是否有 SQL 注入风险" -y

    CI/CD 最佳实践¶

    实践 说明
    始终使用 -y CI 环境无交互终端，不加 -y 会导致授权操作被阻止
    使用 --output-format json 便于程序化解析和结构化日志
    限制 --allowedTools 最小权限原则，只读场景只用 Read,Grep
    使用 --setting-sources project,local 避免加载开发者个人全局配置，确保 CI 行为一致
    设置超时 使用 timeout 命令防止任务无限挂起
    遵守速率限制 批量请求间添加 sleep 延迟
    错误处理 检查退出码，捕获 stderr，避免静默失败

    18. Agent SDK 编程接口¶

    Agent SDK 提供了 TypeScript 和 Python 的编程接口，适合在应用程序中深度集成 CodeBuddy 能力。

    安装¶

    TypeScript：

    npm install @tencent-ai/agent-sdk

    Python：

    pip install codebuddy-agent-sdk

    TypeScript 示例¶

    import { query } from '@tencent-ai/agent-sdk';

async function main() {
  const q = query({
    prompt: '分析项目结构并生成报告',
    options: {
      permissionMode: 'bypassPermissions',
      maxTurns: 20,
      cwd: '/path/to/project',
      model: 'deepseek-v3.1',
    }
  });

  for await (const message of q) {
    if (message.type === 'assistant') {
      for (const block of message.message.content) {
        if (block.type === 'text') {
          console.log(block.text);
        }
      }
    } else if (message.type === 'result') {
      console.log('执行完成，耗时：', message.duration_ms, 'ms');
    }
  }
}
main();

    Python 示例¶

    import asyncio
from codebuddy_agent_sdk import (
    query, CodeBuddyAgentOptions,
    AssistantMessage, TextBlock, ResultMessage
)

async def main():
    options = CodeBuddyAgentOptions(
        permission_mode="bypassPermissions",
        max_turns=20,
        cwd="/path/to/project",
    )
    async for message in query(prompt="分析项目结构", options=options):
        if isinstance(message, AssistantMessage):
            for block in message.content:
                if isinstance(block, TextBlock):
                    print(block.text)
        elif isinstance(message, ResultMessage):
            print(f"执行完成，耗时: {message.duration_ms} ms")

asyncio.run(main())

    细粒度权限控制¶

    通过 canUseTool 回调实现工具级别的权限控制，避免使用 bypassPermissions：

    TypeScript：

    const q = query({
  prompt: '分析项目结构',
  options: {
    canUseTool: async (toolName, input) => {
      const readOnlyTools = ['Read', 'Glob', 'Grep'];
      if (readOnlyTools.includes(toolName)) {
        return { behavior: 'allow', updatedInput: input };
      }
      return { behavior: 'deny', message: `工具 ${toolName} 不允许使用` };
    }
  }
});

    Python：

    from codebuddy_agent_sdk import (
    CodeBuddyAgentOptions, CanUseToolOptions,
    PermissionResultAllow, PermissionResultDeny
)

async def can_use_tool(tool_name: str, input_data: dict, options: CanUseToolOptions):
    read_only_tools = ["Read", "Glob", "Grep"]
    if tool_name in read_only_tools:
        return PermissionResultAllow(updated_input=input_data)
    return PermissionResultDeny(message=f"工具 {tool_name} 不允许使用")

options = CodeBuddyAgentOptions(can_use_tool=can_use_tool)

    CLI 方式 vs SDK 方式对比¶

    维度 CLI 无头模式 Agent SDK
    适用场景 Shell 脚本、CI/CD、管道 应用程序集成、复杂自动化
    调用方式 codebuddy -p "..." query({ prompt: "..." })
    多轮对话 通过 --resume + session_id 原生流式消息支持
    权限控制 --allowedTools 参数 canUseTool 回调函数
    输出格式 --output-format 消息对象流
    进度监控 stream-json 每条消息实时返回
