    CodeBuddy Code 操作文档¶

    CodeBuddy Code 是腾讯云推出的 AI 驱动终端编程助手，提供从代码编写到项目部署的全生命周期 AI 辅助开发能力。

    1. 环境要求与安装¶

    环境要求¶

    • Node.js: 18.20 或更高版本
    • 操作系统: macOS / Linux / Windows

    安装方式¶

    方式一：npm 全局安装（推荐）

    npm install -g @tencent-ai/codebuddy-code

    方式二：原生安装脚本（无需 Node.js）

    # macOS / Linux
curl -fsSL https://copilot.tencent.com/cli/install.sh | bash

# Windows (PowerShell)
irm https://copilot.tencent.com/cli/install.ps1 | iex

    验证安装¶

    codebuddy --version
# 或使用别名
cbc --version

    cbc 是 codebuddy 的简写别名。

    更新¶

    codebuddy update

    2. 首次启动与认证¶

    启动¶

    cd /path/to/your/project
codebuddy

    首次启动时会弹出登录方式选择：

    登录方式 适用场景
    中国站 中国大陆用户
    国际站 海外用户
    企业域名 私有部署，需要自定义服务器地址
    iOA 腾讯内部员工

    使用 ↑↓ 选择，Enter 确认，会打开浏览器完成认证。

    初始化项目上下文（强烈推荐）¶

    > /init

    /init 的作用：

    • 让 CodeBuddy 更准确地理解代码结构和业务逻辑
    • 加快响应速度（避免重复文件扫描）
    • 减少 30-50% 的 Token 消耗
    • 避免提出不兼容的修改方案

    设置响应语言¶

    > /config

    在配置面板中设置 Language 选项，例如选择"简体中文"。

    3. 交互模式使用¶

    基本操作¶

    进入交互模式后，直接输入自然语言即可与 CodeBuddy 对话：

    > 帮我分析这个项目的结构
> 为这个 API 接口写单元测试
> 这段代码有什么潜在的性能问题？

    键盘快捷键¶

    导航与输入¶

    快捷键 功能
    ↑ / ↓ 浏览命令历史
    Tab 自动补全 / 切换思考模式
    Enter 发送消息
    Shift+Enter 换行（需先运行 /terminal-setup）
    \ + Enter 换行（通用方式）
    Ctrl+J 插入换行符
    Ctrl+C 取消当前输入或生成
    Ctrl+D 退出（输入为空时按两次）
    Ctrl+L 清屏（对话历史保留）
    Esc + Esc 回滚对话/代码到之前状态
    Ctrl+G 在外部编辑器中编辑提示词

    权限与模式¶

    快捷键 功能
    Shift+Tab (macOS/Linux) 循环切换权限模式
    Alt+M (Windows) 循环切换权限模式
    Alt+P (macOS) / Alt+P (Windows/Linux) 切换模型

    面板与输出¶

    快捷键 功能
    Ctrl+R 反向搜索命令历史
    Ctrl+O 展开/折叠详细输出
    Ctrl+Y 复制最近一条 AI 回复到剪贴板
    Ctrl+V (macOS/Linux) / Alt+V (Windows) 粘贴（支持文本、图片和图片文件路径）
    Ctrl+B 将当前 Bash 命令移至后台执行

    输入前缀¶

    前缀 功能 示例
    / 斜杠命令 /help
    ! Bash 模式 - 直接运行命令 ! npm test
    @ 文件路径引用 @src/index.ts
    # 记忆快捷方式 - 添加内容到 CODEBUDDY.md # 始终使用 TypeScript

    多行输入方式¶

    方式 快捷键 说明
    反斜杠换行 \ + Enter 所有终端通用
    Shift+Enter Shift+Enter 运行 /terminal-setup 后可用
    控制序列 Ctrl+J 插入换行符
    直接粘贴 粘贴多行文本 自动识别为多行输入

    Bash 模式¶

    在输入前加 ! 可直接运行 Shell 命令，无需 AI 解释或权限审批：

    ! npm test
! git status
! ls -la

    命令及输出会被添加到对话上下文中。

    Vim 编辑模式¶

    通过 /vim 命令启用，支持：

    • 模式切换: Esc 进入 NORMAL 模式，i/a/o 进入 INSERT 模式
    • 导航: h/j/k/l 移动，w/b 跳词，0/$ 行首行尾
    • 编辑: x 删字符，dd 删行，cc 改行，. 重复上次修改

    4. CLI 命令参考¶

    基础命令¶

    codebuddy                    # 启动交互式 REPL
codebuddy "帮我优化这个函数"   # 带初始提示词启动
cbc "查询"                   # 使用别名

    非交互模式（One-Shot）¶

    # 直接查询后退出
codebuddy -p "优化这个 SQL 查询的性能"

# 管道输入
cat error.log | codebuddy -p "分析这些错误日志"

# 需要文件访问时必须加 -y
codebuddy -p "审查 src/utils.js 的代码质量" -y

# 指定输出格式
codebuddy -p "查询" --output-format json

# 限制代理轮次数
codebuddy -p --max-turns 3 "查询"

    会话管理¶

    codebuddy -c                          # 继续最近的对话
codebuddy -c -p "继续我的任务"          # SDK 模式继续对话
codebuddy --resume                    # 交互式选择会话恢复
codebuddy --resume abc123 "继续工作"    # 按 ID 恢复会话

    后台运行¶

    codebuddy --bg "实现登录页面"                  # 后台运行
codebuddy --bg --name feature-x "实现功能"     # 命名后台会话
codebuddy ps                                  # 列出活跃进程
codebuddy logs feature-x                      # 查看日志
codebuddy attach feature-x                    # 附加到后台会话
codebuddy kill feature-x                      # 终止后台会话

    Daemon 守护进程¶

    codebuddy daemon start [--port 8080]  # 启动
codebuddy daemon stop                 # 停止
codebuddy daemon status               # 状态查看
codebuddy daemon restart              # 重启

    常用 CLI 参数¶

    参数 说明
    -p / --print 打印响应后退出（非交互模式）
    -y 跳过权限提示（谨慎使用）
    --model <model> 指定模型
    --agent <name> 指定 agent
    --add-dir <path> 添加额外工作目录
    --plugin-dir <path> 加载本地插件
    --permission-mode <mode> 设置权限模式
    --serve 启动 HTTP 服务
    --worktree [name] 在 git worktree 中运行
    --sandbox 在沙箱中运行
    --verbose 详细日志
    --debug 调试模式

    管道与 Unix 集成¶

    # 管道输入分析
git log --oneline | codebuddy "分析这些提交，找出可能的问题"
cat error.log | codebuddy "帮我分析这些错误日志"

# 输出到文件
cat build-error.txt | codebuddy -p '解释构建错误的根本原因' > output.txt

# 在 CI/CD 中使用
codebuddy -p '检查代码变更中的拼写错误' -y

    6. 常见工作流¶

    理解新代码库¶

    > 给我这个代码库的概览
> 解释这里使用的主要架构模式
> 关键的数据模型有哪些？
> 找到处理用户身份验证的文件

    修复 Bug¶

    > 运行 npm test 时看到错误 <粘贴错误信息>
> 建议几种修复方案
> 更新 user.ts 添加你建议的空值检查

    代码重构¶

    > 在代码库中查找已弃用的 API 使用
> 建议如何重构 utils.js 以使用现代 JavaScript 特性
> 重构 utils.js，保持相同的行为
> 为重构的代码运行测试

    编写测试¶

    > 查找未被测试覆盖的函数
> 为通知服务添加测试
> 为通知服务添加边缘条件的测试用例
> 运行新测试并修复任何失败

    创建 Merge Request¶

    > 总结我最近的代码更改
> 创建一个 MR
> 用更多上下文增强 MR 描述
> 添加关于如何测试这些更改的信息

    使用计划模式¶

    按 Shift+Tab（macOS/Linux）或 Alt+M（Windows）切换到计划模式：

    > 我需要重构身份验证系统以使用 OAuth2。创建详细的迁移计划。
> 向后兼容性怎么办？
> 我们应该如何处理数据库迁移？

    计划确认后再执行，适合多步骤实现和安全分析。

    引用文件¶

    > 解释 @src/utils/auth.js 中的逻辑        # 引用单个文件
> @src/components 的结构是什么？          # 引用目录
> @file1.js 和 @file2.js 的区别是什么？    # 引用多个文件

    处理图片¶

    • 拖放：将图片拖入 CodeBuddy 窗口
    • 粘贴：Ctrl+V 粘贴
    • 路径引用：> 分析这张图片：/path/to/image.png

    扩展思考¶

    在提示词中加入"思考"或"深入思考"关键词启用深度推理：

    > 深入思考实现 OAuth2 认证系统的最佳方法
> 思考这种方法的潜在安全漏洞

    7. 配置系统¶

    配置文件层级¶

    层级 文件路径 用途 Git 跟踪
    用户级 ~/.codebuddy/settings.json 全局个人偏好 否
    项目共享 .codebuddy/settings.json 团队共享的项目标准 是
    项目本地 .codebuddy/settings.local.json 个人的项目覆盖 否

    优先级（高到低）：CLI 参数 > 项目本地 > 项目共享 > 用户级

    常用配置项¶

    {
  "language": "简体中文",
  "model": "gpt-5",
  "agent": "my-reviewer",
  "includeCoAuthoredBy": false,
  "autoCompactEnabled": true,
  "alwaysThinkingEnabled": false,
  "showTokensCounter": false,
  "reasoningEffort": "high",
  "trustedDirectories": ["~/workspace/myproj"],
  "env": {
    "NODE_ENV": "development"
  }
}

    权限配置¶

    {
  "permissions": {
    "allow": ["Bash(npm run lint)", "Bash(npm run test:*)"],
    "ask": ["Bash(git push:*)"],
    "deny": ["WebFetch", "Bash(curl:*)", "Read(./.env)"],
    "additionalDirectories": ["../docs/"],
    "defaultMode": "acceptEdits",
    "disableBypassPermissionsMode": "disable",
    "subagentPermissionMode": "bypassPermissions"
  }
}

    CLI 配置命令¶

    codebuddy config list                        # 列出所有配置
codebuddy config get <key>                   # 获取配置项
codebuddy config set <key> <value>           # 设置项目级配置
codebuddy config set -g <key> <value>        # 设置全局配置
codebuddy config add <key> <value>           # 添加数组配置项
codebuddy config remove <key> [values...]    # 删除配置项

    8. 记忆系统¶

    静态记忆（CODEBUDDY.md）¶

    类型 位置 作用域
    用户记忆 ~/.codebuddy/CODEBUDDY.md 所有项目（个人）
    用户规则 ~/.codebuddy/rules/*.md 所有项目（个人）
    项目记忆 ./CODEBUDDY.md 当前项目（团队共享）
    项目规则 ./.codebuddy/rules/*.md 当前项目（团队共享）
    项目本地记忆 ./CODEBUDDY.local.md 当前项目（个人）

    导入语法¶

    CODEBUDDY.md 支持使用 @ 引用其他文件：

    查看 @README 了解项目概览，@docs/api.md 了解 API 设计。

    条件规则¶

    ---
alwaysApply: false
paths: src/api/**/*.ts
---
# API 开发规则
- 所有 API 端点必须包含输入验证
- 使用标准错误响应格式

    Auto Memory（自动记忆）¶

    CodeBuddy 会自动在会话中保存和检索持久化记忆。

    • 存储位置：~/.codebuddy/memories/{project-id}/
    • 管理命令：/memory
    • 启用/禁用：通过 /config 或 /memory 面板切换

    四种记忆类型¶

    类型 用途
    user 用户角色、目标、偏好
    feedback 用户对 CodeBuddy 行为的纠正
    project 进行中的工作、目标和决策
    reference 外部系统和资源指针

    9. 模型管理¶

    切换模型¶

    方式 命令/操作
    交互面板 /model
    CLI 参数 codebuddy --model <model-name>
    环境变量 export CODEBUDDY_MODEL="model-name"
    快捷键 Alt+P（macOS/Linux/Windows）

    自定义模型配置¶

    在 .codebuddy/models.json 或 ~/.codebuddy/models.json 中配置：

    {
  "models": [
    {
      "id": "deepseek-v4-pro",
      "name": "DeepSeek V4 Pro",
      "vendor": "DeepSeek",
      "url": "https://api.deepseek.com/v1/chat/completions",
      "apiKey": "${DEEPSEEK_API_KEY}",
      "maxInputTokens": 128000,
      "maxOutputTokens": 8192,
      "supportsToolCall": true,
      "relatedModels": {
        "lite": "deepseek-v4-flash",
        "reasoning": "deepseek-v4-pro"
      }
    }
  ],
  "availableModels": ["deepseek-v4-pro", "deepseek-v4-flash"]
}

    重要：url 必须以 /chat/completions 结尾。支持 ${ENV_VAR} 环境变量引用。

    场景自动切换¶

    CodeBuddy 根据任务场景自动切换模型变体：

    变体 用途 环境变量
    lite 轻量任务（提取、摘要） CODEBUDDY_SMALL_FAST_MODEL
    reasoning 深度推理（复杂分析） CODEBUDDY_BIG_SLOW_MODEL

    10. 权限控制¶

    四种权限模式¶

    模式 说明 CLI 参数
    Normal（默认） 根据权限规则逐个确认 -
    Auto-accept Edits 自动批准文件编辑，其他仍需确认 --permission-mode acceptEdits
    Bypass Permissions 跳过所有权限检查 --permission-mode bypassPermissions
    Plan Mode AI 创建计划，等待审批后执行 --permission-mode plan

    切换方式¶

    方式 操作
    键盘切换 Shift+Tab（macOS/Linux）/ Alt+M（Windows）循环切换
    CLI 参数 codebuddy --permission-mode <mode>
    配置文件 settings.json 中的 permissions.defaultMode

    权限规则¶

    {
  "permissions": {
    "allow": ["Bash(npm run test:*)", "Read"],
    "ask": ["Bash(git push:*)"],
    "deny": ["WebFetch", "Read(./.env)"]
  }
}
    • allow：自动批准的操作
    • ask：需要确认的操作
    • deny：拒绝的操作

    11. Skills 技能系统¶

    Skills 是 AI 根据任务需求自动选择的专业能力模块。

    与斜杠命令的区别¶

    特性 Skills 斜杠命令
    触发方式 AI 自动识别调用 用户手动输入
    使用场景 专业领域任务 快捷操作
    可见性 对用户透明 用户主动发起

    创建 Skill¶

    在 .codebuddy/skills/<skill-name>/SKILL.md（项目级）或 ~/.codebuddy/skills/<skill-name>/SKILL.md（用户级）创建文件：

    ---
name: pdf
description: PDF 文档处理专家。当用户需要处理 PDF 相关任务时使用。
allowed-tools: Read, Write, Bash
---
你是一个 PDF 文档处理专家。

## 核心能力
- 提取 PDF 文本内容
- 转换 PDF 格式
- 合并和拆分 PDF

## 工作流程
1. 检查 PDF 文件是否存在
2. 使用适当工具提取内容
3. 根据需求处理

    Frontmatter 字段¶

    字段 说明
    name Skill 名称（默认使用目录名）
    description 描述（帮助 AI 决定何时使用）
    allowed-tools 允许使用的工具白名单
    disable-model-invocation 设为 true 时仅支持手动调用
    context 设为 fork 时在独立子代理中执行
    agent 指定子代理类型（配合 context: fork）

    支持的语法¶

    • $ARGUMENTS：参数占位符
    • @file：文件引用
    • !`command`：Shell 命令执行
    • ${CODEBUDDY_SKILL_DIR}：Skill 目录路径
    • ${ENV_VAR}：环境变量

    /skills

    13. Hooks 钩子系统¶

    Hooks 是在 CodeBuddy 生命周期各阶段自动执行的 Shell 命令。

    事件类型¶

    事件 触发时机 用途
    PreToolUse 工具调用前（可阻止执行） 权限控制、验证、日志
    PostToolUse 工具调用后 自动格式化、后处理
    UserPromptSubmit 用户提交提示词后 输入验证/修改
    Notification 发送通知时 自定义提醒
    Stop 完成响应时 清理、总结
    SubagentStop 子代理完成时 子任务跟踪
    SessionStart 会话开始时 环境设置
    SessionEnd 会话结束时 清理、日志

    配置示例¶

    在 settings.json 中配置：

    {
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | { read f; if echo \"$f\" | grep -q '\\.ts$'; then npx prettier --write \"$f\"; fi; }"
          }
        ]
      }
    ]
  }
}

    交互式设置¶

    /hooks

    选择事件 -> 添加 matcher -> 添加 hook 命令 -> 选择存储位置。

    安全提示：Hooks 使用当前用户环境凭证自动运行，仅注册可信的 Hook。

    14. MCP 集成¶

    MCP（Model Context Protocol）用于集成第三方工具和服务。

    配置文件位置¶

    范围 文件 用途
    用户级 ~/.codebuddy/.mcp.json 全局工具
    项目级 <project>/.mcp.json 项目工具
    本地级 存储在 ~/.codebuddy.json 临时/实验性

    传输类型¶

    STDIO（本地进程）

    {
  "mcpServers": {
    "my-tool": {
      "type": "stdio",
      "command": "python",
      "args": ["-m", "my_mcp_server"],
      "env": { "PYTHONPATH": "/path/to/tools" }
    }
  }
}

    HTTP（远程服务）

    {
  "mcpServers": {
    "api-server": {
      "type": "http",
      "url": "https://api.example.com/mcp",
      "headers": { "Authorization": "Bearer ${API_TOKEN}" }
    }
  }
}

    SSE（Server-Sent Events）

    {
  "mcpServers": {
    "sse-server": {
      "type": "sse",
      "url": "https://api.example.com/mcp/sse",
      "headers": { "X-API-Key": "${API_KEY}" }
    }
  }
}

    CLI 管理¶

    codebuddy mcp add --scope user my-tool -- /path/to/tool arg1 arg2
codebuddy mcp add --scope project --transport http http-server https://example.com/mcp
codebuddy mcp add-json --scope user my-server '{"type":"stdio","command":"/usr/local/bin/tool"}'
codebuddy mcp list                              # 列出所有服务器
codebuddy mcp get my-server                     # 查看详情
codebuddy mcp remove my-server                  # 移除服务器

    环境变量¶

    配置文件支持 ${ENV_VAR} 和 ${ENV_VAR:-default} 语法引用环境变量。

    16. 高级特性¶

    Git Worktree 并行开发¶

    # 创建 worktree
git worktree add ../project-feature-a -b feature-a

# 在不同 worktree 中独立运行 CodeBuddy
cd ../project-feature-a && codebuddy

# 管理 worktree
git worktree list
git worktree remove ../project-feature-a

    CodeBuddy 也内置支持：

    codebuddy --worktree              # 自动创建 worktree
codebuddy --worktree my-feature   # 指定名称
codebuddy --worktree --tmux       # 配合 tmux 使用

    沙箱运行¶

    codebuddy --sandbox                        # 容器沙箱（Docker/Podman）
codebuddy --sandbox https://api.e2b.dev    # E2B 云端沙箱
codebuddy --sandbox --sandbox-new          # 强制创建新沙箱
codebuddy --sandbox --sandbox-kill         # 退出时清理沙箱

    HTTP 服务模式¶

    codebuddy --serve --port 8080

    提供 Web UI、REST API 和 ACP 协议支持。

    子代理系统¶

    使用 --agents 参数动态定义子代理：

    codebuddy --agents '{
  "code-reviewer": {
    "description": "专业代码审查员",
    "prompt": "你是高级代码审查员。",
    "tools": ["Read", "Grep", "Glob"],
    "model": "sonnet"
  }
}'

    或在交互模式中通过 /agents 管理。

    检查点（Checkpoint）¶

    CodeBuddy 支持创建检查点以保存和恢复代码状态。

    无头模式（Headless）¶

    适用于 CI/CD 和自动化场景的非交互模式。

    定时任务¶

    使用 /loop Skill 实现定时任务：

    /loop 5m /standup-report    # 每 5 分钟运行

    或使用 CronCreate 工具调度任务。
